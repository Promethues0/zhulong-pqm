## 1. 产品概述与定位

### 1.1 产品愿景与一句话定位

**愿景**：让任何组织都能在"含密码学相关的量子计算机(CRQC)"到来之前，完成一次有据可查、可被监管核验、不中断业务的后量子密码迁移。

**一句话定位**：烛龙·后量子迁移治理平台(Zhulong PQM, 内部代号 zhulong-pqm)是一款面向私有化/信创交付的**自动化后量子密码(PQC)迁移治理平台**，将"发现→建档→评估→改造→监测"五阶段闭环固化为产品能力，既能用平台内建密码能力直接改造，又能编排烛龙国密网关与外部 HSM/CA 落地迁移，归属烛龙国密零信任统一安全接入产品家族。

> DP-01（命名归属）｜**已决策（2026-06-24）**：正式命名「烛龙·后量子迁移治理平台 / Zhulong PQM」，内部代号 zhulong-pqm，**纳入烛龙国密家族统一品牌**；「烛龙 PQM」短名用于控制台与对外材料。与现有烛龙控制台是否共用入口仍由 DP-10 另议，不在本条范围。

### 1.2 后量子威胁背景

| 要素 | 内容 |
|------|------|
| 破解原理 | Shor 算法可在多项式时间内分解大整数与求解离散对数，**直接攻破 RSA、经典 DH、ECC/ECDSA、SM2** 的数学根基 |
| 资源门槛 | 学界普遍估计需约 **4000 个逻辑量子比特**(对应数百万物理量子比特)即可破解 2048 位 RSA；这是"何时到来"的不确定性，而非"是否到来" |
| 核心威胁模型 | **HNDL(Harvest Now, Decrypt Later)** —— 攻击者今日截获并存储加密流量/密文，待 CRQC 成熟后回溯解密。**对长生命周期数据，威胁此刻已经发生** |
| 标准就绪 | NIST 于 **2024 年正式发布** FIPS 203(ML-KEM 密钥封装)、FIPS 204(ML-DSA 签名)、FIPS 205(SLH-DSA 签名)，PQC 从研究阶段进入工程落地阶段 |
| 对称与哈希 | AES-128 受 Grover 影响安全强度减半(需升 AES-256)；AES-256/SHA-3/SM4/SM3 不受量子显著影响，无需更换 |

### 1.3 为什么是现在（迁移时间窗）

迁移紧迫性由"**数据保密期 + 迁移耗时 + CRQC 到来时间**"三者关系决定(Mosca 定理)。三条驱动同时收紧时间窗：

- **HNDL 驱动**：保密期 ≥7 年的数据(长期合规档案、IoT 设备证书、根 CA 密钥)若今日被采集，未来即被解密。本平台风险模型对 **D2 数据敏感度≥60 且 D3 生命周期≥60** 的资产自动打上 HNDL 标记，意味着这部分资产"等不起"。
- **政策合规驱动**：国际侧 NSM-10 / CNSA 2.0 已设定迁移时间表，NIST NCCoE 推进迁移示范工程；国内侧面向 **等保 2.0、密评、国密局国密+PQC 融合路线**。监管要求企业先"摸清家底(CBOM)"，而手工方式无法持续满足复评。
- **工程耗时驱动**：一次完整迁移参照内置项目计划为 **18 个月、34 个任务、4 阶段**；能源/工控行业还要乘 **1.8 拉伸因子**。倒推之下，迁移必须"现在启动"。

### 1.4 产品化逻辑：4 份手工模板 → 自动化平台

本平台把用户已有的「PQC 迁移方案工具包」4 份手工 Excel/Word 模板，产品化为可持续运行的引擎模块。模板与平台模块的对照关系：

| 工具包模板(手工) | 平台模块(自动化) | 五阶段归属 | 产品化升级点 |
|------------------|------------------|------------|--------------|
| ① 量子脆弱性发现清单(30 检查项/4 层/工具建议) | **发现引擎**(内置 30 条规则库，对接 nmap/sslyze/testssl/Semgrep/Syft 等) | ① 发现 | 人工逐项排查 → 调度自动扫描器，规则库可热更新 |
| ② 密码资产摸底报告模板(CBOM/算法分布/二维矩阵) | **CBOM 建档与摸底报告引擎**(CycloneDX CBOM 自动生成) | ② 建档 | Word 静态报告 → 结构化 CBOM，每季度自动刷新 |
| ③ 五维风险评分模型(D1-D5 加权/预设画像) | **风险评估引擎**(综合分=D1×30%+D2×25%+D3×20%+D4×15%+D5×10%，自动分级 P1-P4) | ③ 评估 | Excel 手算评分 → 引擎自动计算、自动 HNDL 标记与排序 |
| ④ PQC 迁移项目计划(34 任务/4 阶段/甘特) + 迁移验收报告(47 项测试) | **改造编排引擎**(项目计划自适应+行业/规模拉伸因子) + **验收&监测引擎**(自动化测试套件+SLO) | ④ 改造 / ⑤ 监测 | 静态甘特图与人工验收 → 任务编排、设备下发、自动回归测试与持续 SLO 监测 |

闭环主线：① 发现 → ② 建档(CBOM) → ③ 评估(五维) → ④ 改造(内建能力 + 编排外部设备) → ⑤ 监测，构成平台贯穿全文的一条主线。

### 1.5 目标客户与用户画像

**目标行业**（与计划自适应拉伸因子对齐）：金融(×1.3)、政务(×1.0)、医疗(×1.4)、能源/工控(×1.8)、科技(×1.0)。能源工控因 OT 协议+长有效期证书+硬件停机改造，是迁移最重、最需平台编排的行业。

| 安全/密码工程师 | 摸清密码使用点、定算法选型、做混合 KEM 验证 | 发现引擎、CBOM、风险评估、验收套件 |

### 1.6 核心价值主张（对比手工方式痛点）

| 手工 Excel/Word 痛点 | 平台价值主张 |
|----------------------|--------------|
| 30 检查项需人工逐机排查，覆盖不全、易漏 | 调度器批量自动扫描，覆盖可度量、可重复 |
| Excel 五维评分人工填表，口径不一、易出错 | 引擎按固定权重统一计算与分级，自动标 HNDL |
| 报告是某一时点快照，三个月即过期 | CBOM 持续自动更新(默认每季度)，始终反映现网 |

| 国际工具不支持国密、不闭环到设备 | **国密(SM2/SM3/SM4)原生 + 改造执行设备闭环** |

> DP-1.A（竞争路线；关联权威 DP-03 内建vs编排）：是否将"设备改造闭环"作为对外第一卖点，弱化与 IBM 在"发现广度"上的正面比较。**推荐项**：以"国密 + 设备闭环"为主打差异化，发现广度作为达标项而非竞争核心。

### 1.9 产品范围与非目标

**范围内(In Scope)**：密码学发现引擎、CBOM 建档与摸底报告、五维风险评估、迁移项目编排、内建 PQC 改造能力(PQC 就绪 CA、证书生命周期、混合 KEM 反代/配置下发、密码敏捷配置中心)、外部设备/CA/HSM 编排、验收自动化测试与持续监测。

**非目标(Out of Scope，明确不做)**：

| 不做 | 说明/边界 |
|------|-----------|
| 不自研后量子密码算法 | 直接采用 NIST 标准算法(ML-KEM/ML-DSA/SLH-DSA)与国密标准，不发明新算法 |
| 不替代专业 HSM/加密机 | 平台编排 HSM 完成密钥操作，自身不充当密码硬件 |
| 不做通用漏洞扫描器/SIEM | 仅聚焦"密码学相关"脆弱点，通用漏洞/日志分析交由现有安全设施 |
| 不直接改写业务应用源码 | 提供发现(Semgrep 规则)与改造建议，代码改造由业务团队执行 |
| 不承诺破解时间预测 | 不对 CRQC 到来时间给出确定性预测，以 HNDL+保密期驱动决策 |
| 不替代国密合规认证机构 | 输出可审计证据链以支撑密评，但认证结论由权威机构出具 |
| 国密融合已采过渡方案、最终算法待对齐 | 已采 SM2+ML-KEM/SM2+ML-DSA 作为已承诺的过渡方案（非"不做/待定"），最终算法待国密局 PQC 标准定稿后对齐（DP-04 已决策） |

## 2. 总体设计与能力架构

本章给出 zhulong-pqm（烛龙·后量子迁移治理平台）的整体设计蓝图：五阶段闭环的数据流与状态流转、平台能力地图与模块分解、工具包交付物到平台模块的映射、模块间数据契约，以及平台与烛龙家族/外部设备的协同边界。各模块的功能需求细节由后续章节展开，本章只锚定边界与契约。

### 2.1 五阶段闭环总览

平台主线是一条可循环、可回灌的治理闭环：**发现(Discover) → 建档(Inventory/CBOM) → 评估(Assess) → 改造(Remediate) → 监测(Monitor)**，监测阶段的复评信号回灌至发现/评估，形成持续治理而非一次性项目。

```
                    ┌─────────────────────────────────────────────────────┐
                    │                  复评回灌 (re-assess)                  │
                    ▼                                                       │
 ① 发现 ──扫描结果──▶ ② 建档 ──资产对象──▶ ③ 评估 ──评分+计划──▶ ④ 改造 ──工单+验收──▶ ⑤ 监测
 Discover           Inventory/CBOM       Assess              Remediate          Monitor
 (规则库扫描)         (CBOM/资产清单)        (五维评分/分级)       (编排执行/项目计划)   (SLO/告警/CBOM自更新)
```

| 阶段 | 平台动作 | 输入 | 产出（数据对象） | 触达模块 | 退出判据（阶段完成定义） |
|---|---|---|---|---|---|
| ① 发现 Discover | 用 30 项内置规则库扫描 4 层（网络协议/证书密钥/应用数据/供应链）密码学使用点 | 扫描目标/凭据/范围 | 原始扫描结果 `ScanResult` | 发现引擎 | 范围内目标全覆盖，每条结果含算法指纹、量子风险等级（极高/高/中）、原始优先级（P1/P2/P3） |
| ② 建档 Inventory | 去重归并扫描结果为密码资产，生成 CycloneDX CBOM 与摸底报告 | `ScanResult` | 资产对象 `CryptoAsset`、`CBOM`、摸底报告 | CBOM 清单 | 每个资产唯一标识、绑定算法选型建议、QVA 标记，CBOM 可导出 |
| ③ 评估 Assess | 按 D1–D5 五维加权打分、自动分级、HNDL 标记，生成迁移项目计划 | `CryptoAsset` | 评分 `RiskScore`、分级 P1–P4、`MigrationPlan`（34 任务/4 阶段甘特） | 风险评估引擎 + 改造编排 | 全资产有综合分与级别，P1/P2 资产进入计划轨道并标里程碑 |
| ④ 改造 Remediate | 拆解计划为工单，调度平台内建能力或编排外部设备执行，跑验收测试套件 | `MigrationPlan` | 工单 `Ticket`、验收报告 `AcceptanceReport` | 改造编排 + 控制台 + 配置中心 | 工单闭环，47 项级验收套件通过/有条件，P1 资产 100% 覆盖 |
| ⑤ 监测 Monitor | 持续采集握手/性能/告警，监测 SLO，自动更新 CBOM，触发复评 | `AcceptanceReport`、运行遥测 | SLO 指标、告警、`ReassessTrigger` | 监测治理 + CBOM 清单 | SLO 在阈值内（握手失败率<0.1%），复评信号回灌发现/评估 |

### 2.2 平台能力地图与模块分解

平台划分为 8 个一等模块，沿闭环排布；控制台、开放 API、配置中心为横向支撑模块，贯穿全部阶段。

| 模块 | 代号 | 闭环阶段 | 核心职责 | 后续 FR 章节 |
|---|---|---|---|---|
| 发现引擎 | M-DISC | ① | 4 层 30 项规则库扫描调度、第三方工具（nmap/sslyze/testssl.sh/ssh-audit/Semgrep/Syft 等）编排、算法指纹提取 | 第 3 章 |
| CBOM 清单 | M-CBOM | ② | 扫描结果归并去重、CycloneDX CBOM 生成与版本化、摸底报告自动出稿、QVA 标记、资产台账 | 第 4 章 |
| 风险评估引擎 | M-RISK | ③ | D1–D5 五维加权打分、P1–P4 分级、HNDL 自动标记、预设画像库、行业/规模拉伸因子 | 第 5 章 |
| 改造编排 | M-REMED | ③④ | 迁移项目计划生成（34 任务/4 阶段/5 轨道/甘特/里程碑/依赖）、工单拆解、灰度编排 | 第 6 章 |

### 2.4 模块间依赖与数据契约

数据沿闭环单向流动并在监测阶段回灌，核心契约对象如下；每个对象由上游模块产出、下游模块只读消费，禁止跨级直写。

| 数据对象 | 生产者 | 消费者 | 关键字段（契约约束） |
|---|---|---|---|
| `ScanResult` 扫描结果 | M-DISC | M-CBOM | 目标、检查项 ID（1-1 的 30 项之一）、算法指纹、量子风险等级、原始优先级、工具来源 |
| `CryptoAsset` 资产对象 | M-CBOM | M-RISK、M-CONSOLE | 资产唯一 ID、类型、当前算法、QVA 标记、算法选型建议（过渡/长期）、CBOM 引用 |
| `RiskScore` 评分 | M-RISK | M-REMED、M-CONSOLE | 五维向量[D1..D5]、综合分(0–100)、级别 P1–P4、HNDL 标记（D2≥60 且 D3≥60） |
| `MigrationPlan` 迁移计划 | M-REMED | M-CONSOLE、M-API | 任务 ID（34 任务之一）、阶段、轨道、前置依赖、里程碑(MS1–MS4)、拉伸因子、完成% |
| `Ticket` 改造工单 | M-REMED | M-CFG、外部执行体 | 关联资产、目标算法（如 X25519+ML-KEM-768 / SM2+ML-KEM）、执行手段（内建/编排）、灰度阶段 |
| `AcceptanceReport` 验收报告 | M-MON 测试套件 | M-REMED、M-CONSOLE | 测试项通过/有条件/未通过、P1 覆盖率、性能基准实测值、遗留风险登记项 |
| `ReassessTrigger` 复评信号 | M-MON | M-DISC、M-RISK | 触发源（SLO 越界/CBOM 漂移/情报订阅）、影响资产集、建议复评类型 |

数据流主链：`ScanResult → CryptoAsset → RiskScore → MigrationPlan → Ticket → AcceptanceReport → (回灌) ReassessTrigger`。配置中心 M-CFG 横切于工单执行——所有目标算法与策略统一由配置中心下发，保证密码敏捷；控制台 M-CONSOLE 与开放 API M-API 横切于全链，分别提供人机视图与机机接口。

### 2.5 平台与烛龙家族/外部设备的协同边界

平台定位是**治理大脑**：负责发现、评估、决策、编排与验证；**不直接终结密码运算**，运算由执行体完成。执行体分两类——平台内建能力与外部受编排设备。

| 角色 | 实体 | 在闭环中的职责 | 协同接口 |
|---|---|---|---|
| 大脑（决策面） | zhulong-pqm 平台本体 | 发现/建档/评分/出计划/拆工单/验收/监测/复评 | M-API 下发工单、回收验收与遥测 |
| 内建执行体 | PQC 就绪 CA 与证书生命周期、混合 KEM 反代、密码敏捷配置中心 | 直接执行证书签发、TLS 混合握手、配置下发 | M-CFG 策略下发 + M-REMED 工单调度 |
| 外部执行体·烛龙家族 | 烛龙 IPSEC/SSL VPN 国密网关（已支持 `ke1_mlkem` 混合提议） | 接收工单完成 IKEv2 混合 KEM、国密 SM2+ML-KEM 隧道改造 | 编排 API + swanctl/vici 配置回灌验收 |
| 外部执行体·第三方 | HSM/加密机、第三方 CA、OQS-OpenSSL | 密钥材料生成/迁移、证书签发、库级混合 KEM 能力 | 标准化适配器 + 工单/验收契约 |

边界原则：①平台只持有资产元数据与策略，密钥材料不出 HSM；②对烛龙家族走原生编排接口（深度联动），对第三方设备走标准化适配器（最小耦合）；③所有执行结果必须回灌验收套件与 CBOM，未回灌的工单不得置为"已验收"。

> **DP-2.A（产品命名归属；关联权威 DP-01）**：本平台命名为「烛龙·后量子迁移治理平台 / Zhulong PQM」，内部代号 zhulong-pqm，纳入烛龙国密零信任家族。**推荐项：采纳该名**，理由是复用烛龙品牌与黏土橙视觉体系、与现有 IPSEC/SSL VPN 国密网关形成"大脑+执行体"协同闭环；待评审确认后全线锁定。
>
> **DP-2.B（外部设备耦合策略；关联权威 DP-03）**：烛龙家族走原生深度编排、第三方设备走标准化适配器。**推荐项：双轨并行**，优先保障烛龙网关的 `ke1_mlkem` 深度联动，第三方以适配器渐进接入，避免一刀切造成集成债。

### 2.6 资产对象生命周期状态机

每个 `CryptoAsset` 自被发现起进入统一生命周期，状态由闭环阶段驱动；监测阶段的复评使其回到评估态，形成可循环治理。

```
 [发现] ──归并入档──▶ [待确认] ──人工/规则确认──▶ [已建档] ──五维打分──▶ [已评分]
                          │                                                  │
                    （误报/重复→[已忽略]）                         （P4<25 或低危→[监测中]）
                                                                            │（P1/P2/P3 入计划）
                                                                            ▼
 [复评] ◀──复评信号──── [监测中] ◀──验收通过──── [已验收] ◀──套件验证──── [改造中]
   │                                                                        ▲
   └────────────────────回到[已评分]重新分级/重排────────────────────────────┘
```

| 状态 | 进入条件 | 负责模块 | 离开条件（迁出） |
|---|---|---|---|
| 发现 Discovered | 扫描命中规则库检查项 | M-DISC | 归并去重后入档 |
| 待确认 Pending | 归并为候选资产 | M-CBOM | 确认为真 → 已建档；误报/重复 → 已忽略 |
| 已建档 Inventoried | 确认且写入 CBOM | M-CBOM | 触发五维打分 |
| 已评分 Scored | 产出综合分与 P1–P4 级别 | M-RISK | P1/P2/P3 入计划 → 改造中；P4 → 监测中 |
| 改造中 Remediating | 工单已派发并执行 | M-REMED | 验收套件验证完成 |
| 已验收 Accepted | 47 项级套件通过/有条件且回灌 | M-MON/M-REMED | 转入持续监测 |
| 监测中 Monitored | 纳入 SLO 与 CBOM 自更新 | M-MON | 复评信号触发 |
| 复评 Reassessing | SLO 越界/CBOM 漂移/威胁情报 | M-MON→M-RISK | 回到已评分重新分级 |
| 已忽略 Ignored | 判定误报或重复 | M-CBOM | 可被新证据重新激活回待确认 |

状态机约束：①不可跳级——资产不得未经"已评分"直接进入"改造中"；②"已验收"必须满足验收套件回灌（见 2.5 边界原则③）；③"复评"是闭环回灌的唯一入口，确保持续治理而非一次性迁移；④"已忽略"非终态，新扫描证据可将其重新激活，避免规则误判造成永久盲区。

## 3. 密码资产发现（Discover）

本章定义五阶段闭环首环「发现(Discover)」的发现引擎模块需求。发现引擎负责扫描现网密码学使用点，产出标准化的"扫描结果原始记录"，作为第 4 章建档(Inventory/CBOM)的唯一上游数据源。本章只负责"发现并记录原始事实"，不负责打分（第 5 章评估）与改造（第 6 章）。

发现引擎将《PQC迁移方案工具包·文件1-1 量子脆弱性发现清单》的 30 个检查项固化为平台**内置扫描规则库**，配合七类发现方式与统一编排，实现可重复、可增量、可度量覆盖度的自动化发现。

### 3.1 发现方式总览与职责边界

**架构基调（按 DP-02 已决策 2026-06-24）**：发现引擎以 **Agentless 网络扫描为主干（强制能力）**，被动流量分析作为 Agentless 的增强手段；**主机 Agent 为辅助补盲**，仅覆盖网络扫描天然触达不到的本地资产（主机证书/私钥库、SSH 主机密钥、本地密码库版本、本地代码/SBOM 扫描）。因此 M1 主动网络扫描及 M2/M4/M5/M6 等 Agentless 方式构成主干并在 R1 交付（被动流量 M2 为可选）；M3 主机 Agent 为辅助/R2 可选增量。

七类发现方式互为补充，覆盖"网络可见面 + 主机内生面 + 代码供应链面 + 配置/云面 + 人工补录面"。

| 编号 | 发现方式 | 适配检查层 | 主要采集对象 | Agent? | 定位/批次 |
|---|---|---|---|---|---|
| M1 | 主动网络扫描 | L1 | TLS 版本/套件、密钥交换、签名算法、IKE DH 群组、SSH KEX、RDP/Citrix 加密 | Agentless | **主干（强制）·R1** |
| M2 | 被动流量分析 | L1 | 镜像流量中的握手算法、DH 群组、mTLS 状态、IPsec ESP 套件 | Agentless（探针） | Agentless 增强·R1 可选 |
| M3 | 主机 Agent | L2/L3/L4 | 本机证书/私钥库、SSH 主机密钥、本地密码库版本(OpenSSL/mbedTLS)、本地代码/SBOM、KMS 客户端 | Agent | **辅助补盲·R2 可选** |
| M4 | 代码与 SBOM 扫描 | L3/L4 | 代码内密码库调用、依赖 SBOM、库是否支持 ML-KEM | Agentless（CI/仓库） | Agentless 主干·R1（本地仓库补盲可转 M3） |
| M5 | 配置与证书导入 | L1/L2/L4 | VPN/CA/KMS 配置、证书库导出、CRL/OCSP 能力 | Agentless（文件/API） | Agentless 主干·R1 |
| M6 | 云与 KMS API 对接 | L2/L3/L4 | 云证书管理器、KMS 密钥清单与 PQC 路线图、对象存储 SSE | Agentless（API） | Agentless 主干·R1 |
| M7 | 人工录入/访谈补录 | L1-L4 | 自动化无法触达的资产（隔离网、SaaS、外包代码、工控 OT） | 人工 | 人工兜底 |

**FR-3.1 发现方式可独立启用**：平台须支持 M1-M7 七类发现方式各自独立配置、独立运行、独立产出原始记录，且同一资产被多方式命中时须在建档阶段可去重合并（去重键见 FR-3.10）。
- 验收口径：分别仅启用 M1、仅启用 M3 各跑一次任务，均能产出符合 FR-3.10 字段规范的原始记录；同一台主机的"对外 443 证书"被 M1 与 M3 同时发现时，记录中携带可归一的 `asset_fingerprint`，建档侧合并为 1 个资产、保留 2 条证据来源。

**FR-3.2 发现方式与检查层映射可追溯**：每条原始记录须标注其命中的发现方式(M1-M7)与检查层(L1-L4)及具体规则号(见 3.4)。
- 完成定义：任取一条记录可反查"由哪种方式、命中哪条内置规则、属于哪一层"。

### 3.2 主动网络扫描（M1）

**FR-3.2.1 TLS 端点主动探测**：对指定 IP/域名/端口范围，调用 `nmap`、`sslyze`、`testssl.sh` 探测 TLS 版本、密码套件、密钥交换算法(RSA/ECDH/DHE)、证书签名算法(RSA-PKCS1/ECDSA/Ed25519)。
- 验收口径：对一组测试 TLS 端点，能解析出协议版本清单、KEX 算法、签名算法三类字段，并各自映射到规则 R-L1-01/02/03。

**FR-3.2.2 VPN/SSH/IPsec 主动探测**：调用 `ike-scan` 识别 IKEv1/v2 DH 群组(标记 Group2/5/14 为脆弱)、`ssh-audit` 识别 SSH KEX 算法、strongSwan 审计识别 IPsec ESP 套件(标记 AES-128)。
- 完成定义：对一台启用 IKEv2 的烛龙网关，能输出其 DH 群组并对 Group2/5/14 命中 R-L1-04；对一台 SSH 主机能输出 KEX 列表并命中 R-L1-05。

**FR-3.2.3 RDP/Citrix 远程桌面探测**：调用 `Nessus`/`OpenVAS` 等评估器识别 RDP/Citrix 加密配置，命中 R-L1-08。

**FR-3.2.4 探测安全护栏**：M1 须支持速率限制、白名单网段、探测窗口（仅在允许时间段扫描），避免对生产端点造成压力；默认只读探测，不得发送破坏性 payload。
- 验收口径：配置速率上限与时间窗后，超窗任务自动挂起；探测全程不触发被测端连接中断告警。

### 3.3 其余发现方式（M2-M7）

**FR-3.3.1 被动流量分析（M2）**：部署旁路探针，对镜像流量用 Wireshark/sslyze 逻辑识别 TLS 握手算法、DH 群组、微服务 mTLS 状态(对接 Istio 遥测)、IPsec ESP 套件，命中 R-L1-02/04/06/07。
- 完成定义：在镜像口注入一段含 ECDHE 握手的流量，探针能识别 KEX 并生成 M2 来源的原始记录，且不解密载荷（仅读握手元数据）。

**FR-3.3.2 主机 Agent（M3，辅助补盲 / R2 可选）**：按 DP-02，主机 Agent 为 Agentless 主干的辅助补盲手段，仅用于网络扫描天然覆盖不到的本地资产（主机证书/私钥库、SSH 主机密钥、本地密码库版本、本地代码/SBOM），列为 R2 可选增量，非 R1 强制能力。Agent 采集本机服务器证书、代码签名证书、SSH 主机密钥、PGP/GPG 密钥、本机 OpenSSL/LibreSSL/mbedTLS 版本及其 ML-KEM 支持情况、HSM/KMS 客户端配置，命中 R-L2-02/03/05/06、R-L4-01。
- 验收口径：在一台装有 OpenSSL 的主机运行 Agent，能读出库版本并判定"是否支持 ML-KEM"，输出对应 `pqc_ready` 布尔与置信度。

**FR-3.3.3 代码与 SBOM 扫描（M4）**：对接代码仓库/CI，用 `Semgrep` 内置密码规则扫描应用代码密码库调用(R-L3-01)、移动端 APK/IPA 反编译识别内置加密(R-L3-08)；用 `Syft` 生成 SBOM 并识别 OpenSSL/LibreSSL/mbedTLS 是否支持 ML-KEM(R-L4-01)、外包代码密码组件(R-L4-05)。
- 完成定义：对一个含硬编码 RSA 调用的样例仓库，Semgrep 规则命中并生成 M4 记录，附文件路径与行号。

**FR-3.3.4 配置与证书导入（M5）**：支持导入 VPN/CA/KMS 配置文件与证书库导出(PEM/PKCS12/CSV)，离线解析内部根 CA 算法与有效期(R-L2-01)、对外服务器证书清单(R-L2-02)、CRL/OCSP 是否支持新算法(R-L2-07)、HSM 固件 PQC 支持(R-L4-03)。
- 验收口径：导入一份根 CA 证书，能解析出签名算法、有效期，并对 RSA/SM2 根 CA 命中 R-L2-01（极高风险）。

**FR-3.3.5 云与 KMS API 对接（M6）**：通过云厂商/KMS API 拉取证书管理器清单、KMS 密钥材料算法与 PQC 路线图(R-L3-06)、对象存储 SSE 配置(R-L3-05)、数据库静态加密(R-L3-02)、消息队列传输加密(R-L3-04)。
- 完成定义：对接一个 KMS 后能列出密钥算法分布，并标注每个 KMS 是否在 PQC 路线图内。

**FR-3.3.6 人工录入/访谈补录（M7）**：提供表单与批量导入(CSV/Excel)，用于录入自动化无法触达的资产：IoT/嵌入式设备证书及有效期(R-L2-04)、长期合规档案(R-L3-07)、SaaS API TLS 策略(R-L4-04)、工控 OT 协议加密(R-L4-06)、VPN 固件 PQC 升级路径(R-L4-02)、HSM 中长期密钥材料(R-L2-08)、加密备份/归档(R-L3-03)。
- 验收口径：人工补录的记录与自动发现记录字段同构，且 `source=manual`、必须填写 `evidence_note`（访谈来源/责任人）。

### 3.5 扫描任务编排

**FR-3.5.1 任务范围与资产组**：任务须可定义范围（IP 段/域名/CIDR/仓库地址/云账号/资产组标签），并支持按"资产组"复用范围（如"对外业务区""核心 PKI 区""IoT/OT 区"）。
- 验收口径：创建一个绑定"核心 PKI 区"资产组的 M5 任务，仅扫描该组内 CA/KMS 资产。

**FR-3.5.2 定时与触发**：支持手动、定时(cron)、事件触发(如证书库变更、SBOM 更新)；与监测阶段(第 8 章)的"CBOM 持续自动更新"对接。对应《文件1-2》"建议每季度更新"，定时模板须内置季度全量。

**FR-3.5.3 凭据托管**：扫描所需凭据(SSH 密钥、API Token、云 AK/SK、HSM/KMS 访问凭据)须集中托管、加密存储、按任务最小授权引用，凭据明文不得落原始记录。
- 完成定义：原始记录与日志中凭据字段被脱敏；凭据轮换不影响历史记录可读。

**FR-3.5.4 并发与限速**：任务须可配并发度与每目标速率上限，并继承 M1 探测护栏（FR-3.2.4）。

**FR-3.5.5 增量扫描**：支持全量/增量两种模式，增量仅采集自上次扫描后变化的资产（基于 `asset_fingerprint` 与 `last_seen` 差异），降低重复开销。
- 验收口径：同一范围连续两次增量扫描，第二次仅对新增/变更资产产出新证据记录，未变更资产更新 `last_seen` 而不重复建条。

### 3.6 覆盖度与盲区管理

**FR-3.6.1 覆盖度度量**：平台须按"资产组 × 检查层(L1-L4) × 发现方式(M1-M7)"统计覆盖度，并复用《文件1-1》进度仪表板口径展示：总检查项 30、P1 高优 14、极高风险 7 的命中/未命中分布。
- 验收口径：仪表板能显示"30 项中已覆盖 N 项"，并标出哪几条极高风险规则尚未被任何任务覆盖。

**FR-3.6.2 盲区登记**：对自动化无法触达的资产（隔离网、SaaS、无 OTA 工控、外包代码）须登记为"已知盲区"，标注原因与计划补录方式(默认转 M7)，避免被误判为"无风险"。
- 完成定义：一台无 OTA 的工控设备若 M1/M3 不可达，系统不得记为"无资产"，而须生成"盲区待补录"条目并提示走 M7。

**FR-3.6.3 优先级口径对齐**：覆盖度排序须沿用《文件1-1》优先级窗口 P1 立即(0-3 月)/P2 近期(3-6 月)/P3 规划(6-12 月)，优先暴露未覆盖的 P1 项。

### 3.7 Agent vs Agentless 部署选择（决策点）

**DP-3.A（已收敛到顶层 DP-02）｜主机内生面采集架构：Agent vs Agentless。** 本议题已按 DP-02 决策为 **Agentless 主、Agent 辅**：Agentless 网络扫描为主干强制能力（被动流量为增强），主机 Agent 仅作辅助补盲，覆盖本地证书/私钥库、SSH 主机密钥、本地库版本、本地代码/SBOM；R1 仅交付 Agentless，主机 Agent 列为 R2 可选增量。下表权衡保留供部署参考，但选型已不再开放。

| 维度 | Agent（M3 常驻） | Agentless（SSH/API 临时连接） |
|---|---|---|
| 采集深度 | 高（私钥/库版本/本机配置直采） | 中（受远程接口暴露限制） |
| 部署成本 | 高（需安装、信创兼容、升级运维） | 低（仅需凭据与可达性） |
| 实时性/增量 | 强（可常驻监听库/证书变更） | 弱（依赖定时拉取） |
| 适用面 | 服务器集群、关键 PKI/HSM 宿主 | 网络设备、云资产、临时排查 |

- **已采纳（DP-02）**：主干 = Agentless（M1/M2/M4/M5/M6），强制能力、R1 交付；主机 Agent(M3) 为辅助补盲，仅在网络扫描覆盖不到的本地证书/私钥库、SSH 主机密钥、本地库版本、本地代码/SBOM 场景启用，R2 可选增量。Agent 须满足信创/Appliance 私有化交付与国密环境兼容（与烛龙家族一致）。
- 实施细节（非顶层 DP）：核心区是否常驻 Agent、Agent 是否复用烛龙终端客户端通道、最小采集权限边界，留待 R2 设计阶段细化。

## 4. 密码使用点清单与 CBOM（Inventory）

本章定义 zhulong-pqm 五阶段闭环中 ②建档(Inventory/CBOM) 阶段的产品能力。发现引擎(第3章)产出的原始信号在本阶段被归一为唯一的「密码学使用点」实体，沉淀为符合 CycloneDX CBOM 规范的密码资产清单，并向 ③评估(Assess,第5章)输出结构化、可追溯、可比对的数据底座。本章不涉及扫描器规则与发现逻辑(见第3章),也不涉及五维评分(见第5章)。

### 4.1 核心实体：密码学使用点（Cryptographic Usage Point, CUP）

「密码学使用点」是平台的核心资产实体，指现网中一处确定的、产生密码学行为的逻辑位置（一个 TLS 端点、一张证书、一个 IKEv2 提议、一处代码内密码库调用、一份静态加密的数据集等）。CUP 是发现、建档、评估、改造、监测全链路的最小可治理单元，全平台以 CUP 的 `asset_id` 作为外键贯穿。

| system | string | 是 | 所属系统/业务，如「对外门户」「核心交易」 | 资产分组、责任归属 |

| confidence | enum{高,中,低} | 是 | 置信度：多扫描器交叉确认=高；单自动化命中=中；仅人工访谈/推断=低 | 报告标注、复核优先级 |
| hndl_flag | bool | 是 | HNDL(Harvest Now Decrypt Later) 标记，由规则自动置位（见 4.5） | 高风险清单、优先级 |
| data_sensitivity | enum | 否 | 数据敏感度分档（公开/内部/客户隐私/核心机密/国家安全） | D2 评分输入 |
| data_lifecycle | enum | 否 | 数据保密期分档(<1年/1-3/3-7/7-15/>15年) | D3 评分输入、HNDL 判定 |
| status | enum | 是 | CUP 生命周期状态（见 4.4 状态机） | 治理看板、闭环度量 |
| group_tags | array | 否 | 资产分组标签（业务/区域/合规域/自定义） | 分组视图、批量编排 |
| cbom_ref | string(bom-ref) | 是 | 在 CBOM 文档中的 `bom-ref`，关联 CycloneDX 组件 | CBOM 导出、依赖图 |
| version | int | 是 | CUP 记录版本号，每次归一/编辑递增 | 版本化、快照 diff |

**FR-4.1 CUP 实体建模与持久化**：平台应以上表字段建立 CUP 数据模型并持久化，`algorithms` 字段须支持一个 CUP 携带多个算法条目（如同一 TLS 端点的 KEM+SIG+对称+哈希）。
完成定义：能录入并检索一条覆盖全部必填字段的 CUP；`algorithms` 至少支持 KEM/SIG/对称/哈希 四类 family；缺失条件必填字段（如证书类缺指纹）时拒绝入库并给出字段级错误。

### 4.2 对齐 CycloneDX CBOM

平台内部 CUP 模型与 CycloneDX CBOM 标准须建立双向映射，CBOM 为对外交换与合规呈递的标准载体。映射关系：

| CycloneDX CBOM 构件 | 对应平台数据 | 说明 |
|---|---|---|
| component(type=cryptographic-asset) | 一个 CUP | `bom-ref`=`cbom_ref`，name/属性来自 CUP |
| cryptoProperties.assetType | ← CUP 类别（端点=protocol / 证书=certificate / 库调用=algorithm / 密钥材料=related-crypto-material） | 由 CUP 类别映射，非等价于内部 family（二者多对多） |
| cryptoProperties.algorithmProperties.primitive | ← 内部 family（KEM / SIG / 对称 / 哈希） | 内部 family 映射到原语，与 assetType 多对多而非等价 |
| cryptoProperties.algorithmProperties | algorithms{name,mode,param}, key_length | 含 primitive、parameterSetIdentifier、cryptoFunctions |
| cryptoProperties.certificateProperties | cert_fingerprint, cert_validity | 指纹、有效期、签名算法 |
| cryptoProperties.protocolProperties | protocol_endpoint | TLS/IKEv2 版本与套件 |
| dependencies | CUP 间引用关系 | 如证书→签发 CA、TLS 端点→所用证书 |
| evidence / occurrences | evidence[] | 证据出现位置与原始载荷引用 |

**FR-4.2.1 CBOM 序列化与版本声明**：平台应将任意 CUP 集合序列化为合规的 CycloneDX CBOM（JSON），声明 specVersion=1.6（见权威 DP-07 CBOM 标准选型，与 10.3 FR-10.2 的"CycloneDX 1.6 schema 校验"一致）与 `cryptographic-asset` 组件类型，算法以 OID/标准名标识（如 ML-KEM-768、ML-DSA-65、SM2、RSA、ECDH）。
完成定义：导出的 CBOM 通过 CycloneDX schema 校验；随机抽取 5 个 CUP 反序列化回平台后字段无损（round-trip 一致）。

**FR-4.2.2 依赖关系建图**：平台应在 CBOM 中表达 CUP 间依赖（证书→签发 CA、TLS 端点→证书、混合提议→各算法分量），并在前端以依赖图可视化。
完成定义：内部根 CA 被其签发的服务器证书引用时，依赖图能从一张对外证书回溯到根 CA 节点。

### 4.3 多源去重与归一

发现阶段同一对象常被多个扫描器重复命中（如一张证书同时被 sslyze、testssl.sh、openssl x509 抓到）。归一引擎须将其合并为单一 CUP，并保留全部来源作为证据。

**去重主键优先级（自上而下匹配，命中即合并）：**

| 优先级 | 匹配键 | 适用 CUP 类型 |
|---|---|---|
| 1 | cert_fingerprint(SHA-256) | 证书 / CA / 代码签名证书 / 主机密钥 |
| 2 | protocol_endpoint(host+port+proto+sni) | TLS/SSH/IKEv2/IPsec 端点 |
| 3 | 代码位置(repo+file+行号+规则) | Semgrep 代码内密码调用 |
| 4 | SBOM 坐标(purl/组件名+版本) | OpenSSL/liboqs 等供应链组件 |
| 5 | 业务键(system+资源标识)模糊匹配 | 数据集/KMS/对象存储等人工录入 |

**FR-4.3.1 多源合并**：归一引擎按上表主键合并重复命中，合并后 `discovery_source` 与 `evidence` 累积所有来源，`confidence` 按「多扫描器交叉=高」自动升级，算法字段冲突时保留并标记冲突待人工裁决。
完成定义：同一证书被 ≥2 个扫描器命中时，库中仅一条 CUP，其 `discovery_source` 长度 ≥2、`confidence`=高；构造算法字段冲突样本（A报ECDH、B报RSA）时该 CUP 进入「冲突待裁决」队列且不静默丢弃任一来源。

**FR-4.3.2 归一与重扫稳定性**：同一对象在多次扫描中 `asset_id` 保持稳定，不因重扫生成新实体；新增的来源/证据并入既有 CUP 并触发 `version` 递增。
完成定义：对同一现网连续两次全量扫描后，证书类 CUP 数量不翻倍，受影响 CUP 的 `version` 单调递增。

### 4.4 生命周期、版本化与状态机

CUP 自被发现起进入治理生命周期，状态机驱动评估与改造闭环度量。

| 状态 status | 含义 | 进入条件 | 可迁出至 |
|---|---|---|---|
| discovered | 已发现，未归一确认 | 发现引擎首次写入 | inventoried / merged |
| merged | 已被合并入另一 CUP | 命中去重主键 | （终态，指向主 CUP） |
| inventoried | 已建档确认，进入清单 | 归一完成、必填字段齐全 | assessed |
| assessed | 已完成五维评估、定级 | 第5章评分回写 | in_remediation |
| in_remediation | 改造中（含已派工单） | 第6章编排启动 | remediated / risk_accepted |
| remediated | 改造完成、待验收 | 改造执行回报 | verified / reopened |
| verified | 通过验收测试（第8章） | 验收套件通过 | monitored |
| risk_accepted | 风险接受/挂起 | 责任人审批（如 R-003） | in_remediation |
| monitored | 进入持续监测 | 验收通过 | reopened |
| reopened | 复扫发现回退/失效 | 监测告警触发 | in_remediation |

**FR-4.4.1 生命周期状态机**：平台应实现上述状态机，状态迁移须校验合法性、记录操作人与时间戳，非法迁移被拒绝。
完成定义：尝试从 `discovered` 直接迁到 `verified` 被拒绝；每次合法迁移在审计日志留痕（who/when/from→to）。

**FR-4.4.2 记录版本化**：每条 CUP 的关键字段变更（算法、证书、状态、分组）生成新 `version` 与变更记录，支持查看任意历史版本与逐字段 diff。
完成定义：修改某 CUP 的算法参数后能并排查看 v(n-1) 与 v(n) 的字段差异，并可回溯到任一历史版本。

### 4.5 HNDL 自动标记

**FR-4.5 HNDL 判定**：归一/评估阶段，平台应按「`data_sensitivity` ≥ 客户隐私(D2≥60) 且 `data_lifecycle` ≥ 3-7年(D3≥60)」的规则自动置位 `hndl_flag=true`，与第5章 D2/D3 口径一致，并在清单与报告中以醒目标识呈现。
完成定义：构造一条「核心机密 + 7-15年保密期」CUP 自动标 HNDL；「公开 + <1年」不标；HNDL 资产可在清单中一键筛选。

### 4.6 CBOM 快照与差异（季度更新对比）

工具包文件 1-2 要求摸底报告「每季度更新」。平台将其产品化为 CBOM 快照与基线 diff 能力。

**FR-4.6.1 快照与基线**：平台应支持对全量或指定范围的 CUP 集合冻结为带时间戳的 CBOM 快照（基线），并长期留存。
完成定义：可创建命名快照（如 `2026Q2-baseline`）并随时按快照导出 CBOM。

**FR-4.6.2 快照差异对比**：平台应对两个快照做结构化 diff，按 CUP 维度输出「新增 / 移除 / 算法变更 / 证书续期或更换 / 状态变化 / 风险等级变化」，并汇总算法分布的环比变化。
完成定义：在基线后新增一张 RSA 证书并将某端点从 ECDH 改为 X25519+ML-KEM-768，diff 报告分别归类为「新增」与「算法变更（迁移进展）」，并反映 RSA 占比下降。

### 4.7 证据链与可追溯

**FR-4.7 证据链**：每个 CUP 的每条结论（算法判定、HNDL 标记、风险定级）须可下钻到支撑证据（扫描原始输出、配置片段、Semgrep/Syft 条目、访谈记录），证据载荷以哈希固化防篡改，并记录采集时间与来源工具。
完成定义：从清单中任一高风险 CUP 一键下钻到其原始扫描输出与对应 `discovery_source.rule_id`（关联第3章发现规则）；证据哈希校验失败时给出告警。

### 4.8 导入导出与互操作

**FR-4.8 导入导出**：平台应支持 CBOM(CycloneDX JSON) 与清单(CSV) 的导入与导出；导入外部 CBOM 时经归一引擎并入现有清单（去重不重复建实体）；导出支持按层级/系统/分组/风险等级过滤。
完成定义：导出的 CSV 含全部清单字段并可被 Excel 正确解析；导入一份外部 CBOM 后，与既有 CUP 重复者命中去重、新增者建档，导入结果给出「新增/合并/冲突」计数。

### 4.9 算法分布统计

**FR-4.9 算法分布统计**：平台应基于 CBOM 实时统计算法分布，至少覆盖 RSA / ECDSA / ECDH / SM2 / AES / SHA（及 ML-KEM、ML-DSA 等 PQC 算法）的使用点数量与占比，支持按层级/系统/分组下钻，并区分「量子脆弱（Shor 可破：RSA/ECDH/ECDSA/SM2）/ 量子削弱（AES-128）/ 量子安全（AES-256/SHA-3/SM4/ML-KEM/ML-DSA）」三类汇总。
完成定义：统计页给出各算法族 CUP 数与百分比，三类汇总占比合计 100%；下钻到 L1 时只统计网络与协议层 CUP。

### 4.10 摸底报告一键生成

将工具包文件 1-2《密码资产摸底报告》产品化为从 CBOM 自动渲染的报告引擎，消除手工撰写。

**FR-4.10.1 报告自动渲染**：平台应从当前 CBOM/快照一键生成摸底报告，自动填充以下章节，数据全部来自 CUP 清单：

| 报告章节 | 数据来源 | 渲染口径 |
|---|---|---|
| 执行摘要 | 全量统计 | 总 CUP 数、量子脆弱占比、HNDL 资产数、P1 高风险数 |
| 威胁背景 | 模板常量 | Shor + HNDL（约 4000 逻辑量子比特） |
| 调查范围/方法 | discovery_source/layer | 实际覆盖的层级与扫描工具清单 |
| 算法分布总览表 | FR-4.9 统计 | RSA/ECDSA/ECDH/SM2/AES/SHA 占比 |
| 分域结果 | 按 L1-L4 分组 | 各层 CUP 数、风险分布 |
| 高风险资产清单 | hndl_flag + 评估等级 | P1/极高风险 CUP 列表 |
| 算法选型建议 | 选型映射表（下） | 现算法→过渡(Hybrid)→长期目标 |
| 资源与下一步 | 状态机统计 | 待改造数、分批建议(0-6/6-12/12-18月) |

算法选型建议映射表（内置，随报告自动套用，对齐工具包文件 1-2）：

| 类别 | 现状算法 | 过渡（Hybrid） | 长期目标 |
|---|---|---|---|
| 密钥封装 | ECDH / RSA-OAEP | X25519 + ML-KEM-768 | ML-KEM-768 |
| 数字签名 | ECDSA / RSA-PSS | ECDSA + ML-DSA-65 | ML-DSA-65 |
| 国密融合 | SM2 / SM4 | `SM2+ML-KEM-768（KEM）/ SM2+ML-DSA-65（SIG）`（DP-04 已决策，过渡锁定） | 待国密 PQC 标准对齐 |
| 对称加密 | AES-128 | 直接升级 AES-256（无混合期） | AES-256 |
| 哈希 | SHA-256 / SM3 | — | 不变 |

完成定义：对一份含上述各类算法的 CBOM 一键生成报告，八个章节全部有数据填充；算法选型建议表按 CUP 实际命中的现状算法自动给出对应过渡/长期项（如清单含 ECDH 则建议行出现 X25519+ML-KEM-768）。

**FR-4.10.2 报告模板可配置与导出 docx**：报告模板（章节启用/顺序/抬头/术语表 PQC/ML-KEM/ML-DSA/HNDL/Hybrid KEM/CBOM/HSM/QVA）可配置，支持导出 docx（及 PDF），样式采用烛龙「Claude 黏土橙」暖色系（accent #B4552D/#DB855C）。
完成定义：关闭「威胁背景」章节后重新生成，导出 docx 不含该章；导出文件可在 Word 正常打开、目录与表格完整、配色符合规范。

> **DP-4.A（决策点）报告导出技术路线**：docx 渲染存在两条路线 —— ①后端 Go 模板引擎直接生成 docx（一致性强、依赖少）；②对接外部文档能力（如飞书 docx / 第三方排版服务，样式更丰富但引入外部依赖）。**推荐项：路线①** 后端 Go 直生成 docx，满足私有化/信创/Appliance 离线交付约束，外部排版作为可选增强。

## 5. 风险评估与优先级（Assess）

本章定义烛龙·后量子迁移治理平台（zhulong-pqm）五阶段闭环中的第三阶段——风险评估引擎（Assess Engine）。该引擎消费阶段②建档（CBOM）输出的密码使用点清单，对每个资产执行五维风险建模、综合评分、HNDL 标记与 P1–P4 分级，产出供阶段④改造（Remediate）调度的优先级队列与风险登记册。引擎所有维度、权重、档位、阈值默认值均来自工具包文件 2-1/2-2 的五维风险评分模型，并支持管理员在密码敏捷（Crypto Agility）原则下进行配置化覆盖。

### 5.1 五维风险评分模型规范

综合风险分（满分 100）按加权求和计算：

```
综合分 = D1×30% + D2×25% + D3×20% + D4×15% + D5×10%
```

各维度满分均为 100，权重之和恒等于 100%。五个维度的档位与分值定义如下表（即引擎评分基准表，所有取值为系统默认值，不得在运行时随意篡改，变更须经 5.7 配置流程）。

| 维度 | 权重 | 含义 | 档位与分值 |
|---|---|---|---|
| **D1 算法脆弱性** | 30% | 算法被量子计算攻破的可能性 | 10=不受量子影响(AES-256/SHA-3/SM4)；40=对称强度减半(AES-128)；70=经典ECC类被Shor破(ECC/ECDSA/EdDSA)；90=Shor直接破(RSA/经典DH/SM2)；100=已知脆弱(弱RSA≤1024/DES/RC4) |
| **D2 数据敏感度** | 25% | 资产承载数据的机密性等级 | 10=公开；30=内部；60=客户隐私/合同/财务；85=商业核心机密/IP/重要个人信息；100=国家安全/关键基础设施/军事 |
| **D3 数据生命周期** | 20% | 数据需保密的时长（HNDL 关键维度） | 10=<1年；30=1-3年；60=3-7年；85=7-15年；100=>15年/永久 |
| **D4 迁移复杂度** | 15% | 改造该资产的工程难度 | 10=配置变更；35=软件升级；60=系统改造/协议栈重构；85=硬件替换(HSM)；100=硬件+固件+停机(工控IoT) |
| **D5 暴露面** | 10% | 攻击者可触达资产的程度 | 10=完全内网隔离；40=内网可访问；70=半公开合作伙伴；90=互联网公开；100=公开且有大规模录制证据 |

**FR-5.1** 引擎须实现上述加权公式与五档分值表。验收：对任意五维向量 `[D1,D2,D3,D4,D5]`，引擎计算结果与手工加权值误差为 0；以内部根 CA 向量 `[90,100,100,85,10]` 输入，原始浮点 `raw_score=85.75`（90×0.3+100×0.25+100×0.2+85×0.15+10×0.1=27+25+20+12.75+1=85.75），按 FR-5.2 四舍五入(round-half-up)得综合分 **86**（85.75 留存于审计字段 `raw_score`）。

**FR-5.2** 取整口径须全平台统一并显式声明：综合分按四舍五入取整为整数用于分级与展示，原始浮点值（如 85.75）留存于审计字段 `raw_score`。验收：预设画像计算出的整数分与 5.5 预设表完全一致（根 CA=86、SSL VPN=82、对外 TLS=41、长期档案=75、IoT=75、数据库=52、代码签名=74）。

### 5.2 分级阈值、处理时限与 HNDL 标记

**FR-5.3** 引擎按综合整数分映射 P1–P4 等级与处理时限，规则如下表（与阶段④改造的轨道排期、阶段①发现的优先级口径一致）：

| 等级 | 综合分区间 | 风险定性 | 处理时限 | 改造阶段对应 |
|---|---|---|---|---|
| **P1** | ≥75 | 极高 | 立即（0-3 月） | 阶段二·先行批次 |
| **P2** | 50-74 | 高 | 近期（3-6 月） | 阶段三·规模扩展 |
| **P3** | 25-49 | 中 | 规划（6-12 月） | 阶段三/四 |
| **P4** | <25 | 低 | 持续监控 | 阶段五·监测 |

验收：边界值精确归类——75→P1、74→P2、50→P2、49→P3、25→P3、24→P4；区间无重叠无空洞。

**FR-5.4** HNDL（Harvest Now Decrypt Later）自动标记：当且仅当 `D2≥60 且 D3≥60` 时，资产 `hndl_flag=true`。该标记独立于综合分，仅依赖 D2、D3 取值，用于在看板与登记册中突出"现在被窃听、未来被解密"的高危资产。验收：向量 `[70,60,60,10,40]` 标记为真；`[90,85,30,85,90]`（D3=30<60）标记为假；`[10,85,100,10,10]`（D2=85,D3=100）标记为真。

### 5.3 维度取值的自动推导规则（核心能力）

**FR-5.5** 引擎须从阶段①发现/阶段②CBOM 的结构化数据自动推导每个资产的五维取值，无需人工逐项打分。推导依据如下规则表（命中即赋值；多条命中取该维度最高分；无证据命中则置 `null` 并进入人工补全队列）：

| 维度 | 输入信号（来自发现/CBOM 字段） | 推导赋值 |
|---|---|---|
| **D1** | 算法=弱RSA≤1024 / DES / RC4 | 100 |
| | 算法=RSA / 经典DH / SM2 | 90 |
| | 算法=ECC / ECDSA / EdDSA / Ed25519 | 70 |
| | 算法=AES-128 | 40 |
| | 算法=AES-256 / SHA-3 / SM4 / SM3 | 10 |
| **D2** | 数据分类标签=国家安全/关键基础设施/军事 | 100 |
| | 标签=核心机密/IP/重要个人信息 | 85 |
| | 标签=客户隐私/合同/财务 | 60 |
| | 标签=内部 | 30 |
| | 标签=公开 / 无标签默认 | 10 |
| **D3** | 证书/密钥有效期 或 数据留存期 >15年/永久 | 100 |
| | 7-15年 | 85 |
| | 3-7年 | 60 |
| | 1-3年 | 30 |
| | <1年 | 10 |
| **D4** | 资产类型=工控/IoT（固件+停机） | 100 |
| | 密钥材料驻留 HSM（需硬件替换） | 85 |
| | 协议栈/系统级改造（如 VPN 网关、mTLS 网格） | 60 |
| | 仅软件升级（库/中间件） | 35 |
| | 代码签名/TSA 升级（流程/配置变更档） | 10 |

| **D5** | 暴露面=互联网公开 且 有大规模流量录制证据 | 100 |

| | 半公开/合作伙伴可达 | 70 |
| | 内网可访问 | 40 |
| | 完全内网隔离 | 10 |

验收：给定一条 CBOM 记录 `{algorithm: "RSA-2048", data_class: "核心机密", validity_years: 20, asset_kind: "code_signing", exposure: "internet"}`，引擎自动推导 `[90,85,100,10,90]`；D4 因 `code_signing`（代码签名/TSA 升级）属"流程/配置变更"档，映射 D4=10，推导结果须可追溯到具体命中规则 ID（每条规则有稳定 `rule_id`，写入 `derivation_trace`）。

**FR-5.6** 推导可解释性：每个维度取值须附带 `derivation_trace`，记录命中的 `rule_id`、源 CBOM 字段、原始值与赋值理由。验收：任取一资产，可在详情页展开看到五维各自的推导链路；`null` 维度明确标注"待人工补全"原因（无对应字段/字段为空/字段值未匹配任何规则）。

### 5.4 人工复核、覆盖与批量评分

**FR-5.7** 人工覆盖（Override）：评估专家可对任一维度取值或 HNDL 标记做手工覆盖，覆盖值优先于自动推导值参与综合分计算。覆盖须记录操作人、时间、原值、新值、理由，进入审计日志。验收：覆盖后综合分实时重算并刷新等级；自动推导值仍保留为 `auto_value` 可一键回退；未填理由不允许提交。

**FR-5.8** 复核状态机：每个资产评估记录具备状态流转 `待评估 → 自动已评 → 待复核 → 已确认 / 已覆盖 → 已确认`，仅"已确认"状态的评估可进入阶段④改造队列。

| 状态 | 进入条件 | 可执行动作 | 后继状态 |
|---|---|---|---|
| 待评估 | CBOM 新建/变更 | 触发自动评分 | 自动已评 |
| 自动已评 | 引擎完成五维推导 | 提交复核 / 直接确认 | 待复核 / 已确认 |
| 待复核 | 存在 null 维度或被抽中复核 | 补全/覆盖/确认 | 已覆盖 / 已确认 |
| 已覆盖 | 任一维度被人工改写 | 确认 | 已确认 |
| 已确认 | 复核通过 | 重新评估（CBOM 再变更时）| 待评估 |

验收：含 `null` 维度的资产不能直接跳到"已确认"，必须先补全。

**FR-5.9** 批量评分：支持对全量 CBOM 或按域（TLS/VPN/PKI-KMS/应用/数据存储/供应链）、按发现层级（L1-L4）筛选后批量触发评分，异步任务化执行并返回进度。验收：对 ≥1 万条 CBOM 记录批量评分在单节点 5 分钟内完成；任务可断点续跑；结果可导出 CSV/CycloneDX 扩展字段。

## 6. 迁移改造与编排（Remediate）

本章定义五阶段闭环第④环「改造」的实现。它是 zhulong-pqm 区别于纯分析类竞品（如 InfoSec Global AgileSec Analytics、SandboxAQ）的核心：不止给出建议，还能**真正执行**后量子改造。

**主次定位（按 DP-03 已决策 2026-06-24）**：改造主线 = **编排外部设备**（章 6.3）——烛龙 IPSEC/SSL VPN 国密网关下发 ke1_mlkem 混合 IKEv2 提议、HSM/加密机生成与托管混合根 CA 密钥、第三方 CA 签发混合/ML-DSA 证书；**平台内建密码能力（章 6.2）重定位为"编排的补充与兜底"，不追求自建完整密码基础设施**。R1 改造先打通"编排烛龙网关混合 KEM 下发 + HSM 混合根 CA"一条龙，内建 CA/反代后置到 R2。改造执行体两类并存但以编排为主，并由「改造剧本(playbook)库」（章 6.4）、「项目计划引擎」（章 6.5）、「工单与变更/回退」（章 6.6）、「算法选型建议引擎」（章 6.7）统一编排与治理。改造任务的输入来自评估引擎（章 5）输出的优先级队列（P1≥75 / P2=50–74 / P3=25–49 / P4<25）。

### 6.1 改造执行体模型与总体架构

平台把所有改造动作抽象为「步骤(step)→执行体(executor)→交付物(artifact)→验收点(gate)」四元组。一个执行体是能落地改造的最小能力单元，分内建型与编排型两类，登记在执行体注册表中。

| 执行体ID | 类别 | 能力 | 对接方式 |
|---|---|---|---|
| EX-CA | 内建 | PQC就绪CA：签发混合/ML-DSA证书、双签名 | 平台内 Go 服务 |
| EX-PROXY | 内建 | 混合KEM反向代理 / 配置下发(Nginx+liboqs) | 平台内 + Agent 下发 |
| EX-AGILITY | 内建 | Crypto-Agility 算法配置中心 | 平台内策略库 |
| EX-IPSEC | 外部编排 | 烛龙 IPSEC 国密网关，下发 ke1_mlkem 混合IKEv2提议 | VICI / REST |
| EX-SSLVPN | 外部编排 | 烛龙 SSL VPN 网关，混合KEM + 客户端强制升级 | REST |
| EX-HSM | 外部编排 | HSM/加密机，混合根CA密钥生成与托管 | PKCS#11 |
| EX-3CA | 外部编排 | 第三方CA（签发对外服务器证书） | ACME / EST |
| EX-OQS | 外部编排 | OQS-OpenSSL / Nginx 灰度节点 | Agent / SSH |

FR-6.1：执行体注册表必须为每个执行体记录类别、能力标签、健康状态、可改造资产类型、所需凭据句柄。验收口径：注册≥上表 8 类执行体；任一执行体下线时，引用它的进行中任务自动转入「阻塞」态并告警；执行体健康探测周期≤60s。

FR-6.2：每个改造任务执行前必须做「预检(preflight)」——校验执行体可达、目标资产存在于 CBOM（章 4）、当前算法与评估结论一致。验收口径：预检失败的任务不得进入「执行中」态，且在工单上留痕失败原因。

> DP-6.A（执行体扩展策略；与权威 DP-03 对齐）：按 DP-03，**执行体以外部设备编排为主**（烛龙网关/HSM/第三方 CA），内建执行体为补充与兜底；本议题仅决定外部执行体是否开放第三方插件 SDK（让用户自接非烛龙网关/国产 HSM）。推荐项：v1 仅内置上表 8 类适配器（其中外部编排型为改造主线），v2 开放基于 gRPC 的执行体插件协议。

### 6.2 平台内建改造能力（A 类执行体，编排的补充与兜底 / 内建 CA·反代后置 R2）

> 定位（DP-03）：内建密码能力是**编排的补充与兜底**，不追求自建完整密码基础设施。R1 改造主线由章 6.3 编排外部设备承载（烛龙网关混合 KEM 下发 + HSM 混合根 CA 一条龙）；本节内建 EX-CA（PQC 就绪 CA）/EX-PROXY（混合 KEM 反代）后置到 R2，仅在外部设备无法覆盖时作兜底。

FR-6.3（EX-CA · PQC就绪CA与证书生命周期）：平台内建 CA 必须支持三种签发模式——纯 ML-DSA-65、混合(ECDSA+ML-DSA-65 双签名/双证书)、经典(向后兼容)，并管理证书全生命周期（签发/续期/吊销/CRL·OCSP）。验收口径：① 能签发一张通过 `openssl verify` 的混合证书链（根CA→中间CA→叶证书）；② 双签名证书经 ML-DSA 验签与 ECDSA 验签均通过；③ 吊销后 OCSP 在线查询返回 revoked。

FR-6.4（EX-PROXY · 混合KEM反代/配置下发）：平台提供混合 KEM 终止能力，可作反向代理直接承载 X25519+ML-KEM-768 握手，或向目标 Nginx/网关下发 liboqs 配置。验收口径：`openssl s_client -groups X25519MLKEM768` 返回 `Server Temp Key: X25519MLKEM768`；旧客户端 `curl --curves X25519` 仍可回退握手成功。

FR-6.5（EX-AGILITY · Crypto-Agility 算法配置中心）：提供集中的算法策略库，可按资产组/域定义「当前算法集→目标算法集」，一次切换、批量下发、版本化、可回滚。对应项目计划阶段四「Crypto Agility 算法配置中心化」任务。验收口径：① 修改一条算法策略可联动下发到 ≥1 个 EX-PROXY/EX-IPSEC 实例；② 每次策略变更生成不可变版本号并支持一键回退到上一版本；③ 策略至少覆盖 KEM、签名、对称、哈希四类算法位。

FR-6.6（密钥材料溯源）：内建能力产生的混合会话密钥必须可溯源，记录其由 `X25519 ∥ ML-KEM-768` 的共享密钥**拼接(concatenation)后经 TLS1.3 密钥调度 HKDF-Expand-Label(HKDF-SHA-256)派生**，context/label 绑定 `hybrid-kem-v1`（对齐 draft-ietf-tls-hybrid-design）。验收口径：监测/审计（章 7）可查询任一会话的密钥合并溯源记录，与验收测试套件（章 8）口径一致。

### 6.3 外部设备编排（B 类执行体，改造主线）

> 定位（DP-03）：本节为改造**主线**。R1 优先打通"编排烛龙网关混合 KEM 下发（EX-IPSEC/EX-SSLVPN）+ HSM 混合根 CA（EX-HSM）"一条龙；第三方 CA（EX-3CA）与 OQS 节点（EX-OQS）随轨道铺开。执行体以外部设备为主，内建能力（章 6.2）仅作补充与兜底。

FR-6.7（EX-IPSEC/EX-SSLVPN · 烛龙网关编排）：平台向烛龙 IPSEC/SSL VPN 国密网关下发混合提议并管理客户端升级。IPSEC 侧下发 `ke1_mlkem` 混合 IKEv2 提议（MLKEM_768+X25519），SSL VPN 侧推送混合 KEM 配置并强制客户端升级（目标升级率>95%）。验收口径：① 下发后 `swanctl --list-sas` 显示 `ke-method: MLKEM_768+X25519`；② 客户端版本矩阵 v5.2 新版强制、v4.x 旧版享 30 天宽限期；③ 升级率仪表达 >95% 标记轨道完成。

FR-6.8（EX-HSM · 混合根CA密钥）：平台编排 HSM/加密机在硬件内生成并托管混合根CA密钥对，私钥不出 HSM。验收口径：根CA密钥对在 HSM 内生成（PKCS#11 句柄可查），签发动作经 HSM 完成；对应阶段二轨道①步骤「HSM 内生成混合根CA密钥对」。

FR-6.9（EX-3CA / EX-OQS · 第三方CA与OQS节点）：平台可编排第三方CA签发对外服务器证书（ACME/EST），并对 OQS-OpenSSL/Nginx 灰度节点下发配置与采集状态。验收口径：能对一个 Nginx+liboqs 节点完成「下发→灰度→采集握手指纹」闭环，灰度比例与剧本（章 6.4）一致。

> DP-6.B（国密PQC融合执行）：在国密PQC标准落地前，SM2+ML-KEM / SM2+ML-DSA 融合改造由 EX-IPSEC（GM strongSwan）先行承载，还是等标准。推荐项：网络层(IPSEC/SSL VPN)用 GM strongSwan 混合提议先行，证书层 SM2+ML-DSA 标记为「待国密PQC标准」占位，不强推生产。

平台把工具包文件 3-1 的迁移任务沉淀为可复用「改造剧本库」。每条剧本绑定一条轨道，由「步骤→执行体→交付物→验收点」组成；剧本可被项目计划引擎（章 6.5）实例化。

FR-6.14（工单化）：每个改造步骤落地为一张工单，承载执行体、目标资产、计划/实际时间、完成%、验收点状态、责任人。工单状态机如下。

| 状态 | 进入条件 | 可迁移至 |

FR-6.16（回退机制）：每条剧本必须声明回退动作；验收未过或灰度失败率超阈值时可一键回退到改造前配置。验收口径：① PB-TLS 任一灰度档失败可回退该档至改造前（旧客户端不受影响）；② EX-AGILITY 策略变更可回退到上一不可变版本（FR-6.5）；③ 回退动作全程留痕并可生成遗留风险登记项（R-00x 风格，对接章 8）。

### 6.7 算法选型建议引擎

引擎为每个资产/算法位给出「过渡(混合)→长期目标」推荐，数据基线为工具包文件 1-2 选型建议表，并对国密路线给出占位结论。

| 算法位 | 当前 | 过渡（混合） | 长期目标 |
|---|---|---|---|
| 密钥封装 KEM | ECDH / RSA-OAEP | X25519+ML-KEM-768 | ML-KEM-768 |
| 数字签名 | ECDSA / RSA-PSS | ECDSA+ML-DSA-65 | ML-DSA-65 |
| 国密融合 | SM2 / SM4 | `SM2+ML-KEM-768（KEM）/ SM2+ML-DSA-65（SIG）`（DP-04 已决策，过渡锁定） | 待国密 PQC 标准对齐 |
| 对称加密 | AES-128 | 直接升级 AES-256（无混合期） | AES-256 |
| 哈希 | SHA-256 / SM3 | 不变 | 不变 |

FR-6.17：建议引擎须对每个 CBOM 资产位输出三段式推荐（过渡/长期/迁移批次 0-6 / 6-12 / 12-18 月），并与评估优先级（章 5）和剧本（章 6.4）联动。验收口径：① 对一个 SM2 资产输出过渡=SM2+ML-KEM、长期=「待国密PQC标准」并标占位不强推生产；② 对 AES-128 资产输出对称升级 AES-256；③ SHA-256/SM3 标注「不变」；④ 推荐可一键映射到对应剧本生成任务。

FR-6.18（融合占位治理）：标记为「待国密PQC标准」的项不得进入生产强制改造，仅可在网络层（EX-IPSEC，GM strongSwan）以混合提议先行验证（DP-6.B）。验收口径：证书层 SM2+ML-DSA 项在计划中显示为占位/观察态，不计入 MS2 强制完成判定。

> DP-6.C（重加密窗口）：PB-DATA 长期数据批量重加密对在线业务的影响窗口策略。推荐项：能源工控等高敏行业默认停机窗口重加密，其余行业默认在线滚动重加密并以 820MB/s 基准排产，超期自动告警。

## 7. 验收与持续监测（Verify & Monitor）

本章定义五阶段闭环中改造阶段（④ Remediate）的出口质量门与运行态长期治理能力，即闭环第五阶段「⑤ 监测(Monitor)」的平台化落地。验收子系统把工具包文件4的47项测试沉淀为可重复执行的自动化套件，作为单个资产或批次从「改造中」流转到「已完成」的强制 Gate；持续监测子系统则把验收时建立的基线转为长期 SLO 守护、密码漂移检测与复评触发，确保已迁移资产不发生回退、未迁移风险不被遗忘。

### 7.1 验收测试套件（Verification Test Suite）

验收套件以「资产/批次」为执行对象，按四大类组织。整套基线为文件4第二阶段实测的 47 项（44 通过 / 3 有条件 / 0 未通过），P1 资产覆盖率 16/16=100%。验收 Gate 锁定口径（与 1.8 KPI、FR-7.6 一致）：「通过」+「有条件通过」= 47/47 且「未通过」=0；其中纯「通过」≥44（基线 44 通过 / 3 有条件）；有条件项须挂遗留风险登记方计入达标。每个测试用例为声明式定义（探针命令 + 期望断言 + 判定口径），由平台调度执行并自动归档证据（命令输出、抓包、握手元数据）。

**FR-7.1（协议层验证用例库）** 平台内置以下 6 类协议层断言式测试用例，参数与期望值锁定如下表。

| 用例号 | 测试项 | 探针/命令 | 期望断言（通过条件） |
|---|---|---|---|
| V-PROTO-01 | TLS 混合 KEM 握手 | `openssl s_client -groups X25519MLKEM768` | 响应含 `Server Temp Key: X25519MLKEM768`，握手成功 |
| V-PROTO-02 | 旧客户端回退 | `curl --curves X25519` | 服务器协商出纯 X25519，连接成功（向后兼容） |
| V-PROTO-03 | 证书链验证 | `openssl verify` | 证书链至混合根 CA 完整可信，无断链 |
| V-PROTO-04 | IKEv2 混合 KEM | `swanctl --list-sas` | SA 的 `ke-method` 为 `MLKEM_768+X25519` |
| V-PROTO-05 | OCSP 在线验证 | OCSP responder 查询 | 证书状态返回 `good`，响应签名有效 |
| V-PROTO-06 | ML-DSA 双签名 | 代码签名验签 | `RSA + ML-DSA` 双签名均验证通过 |

完成定义：6 类用例全部对目标资产返回 PASS；任一返回 FAIL 时该资产不得进入「已完成」状态。验收口径：探针实际输出与期望断言逐字段比对，证据文件（含原始 stdout 与时间戳）落库可回溯。

**FR-7.2（兼容性矩阵验证）** 平台维护并执行客户端兼容性矩阵，覆盖浏览器 / 移动 / VPN 客户端，判定 PASS / 有条件 PASS / FAIL 三态。

| 客户端 | 期望行为 | 判定 |
|---|---|---|
| Chrome 131+ / Firefox 132+ | 协商混合握手 X25519MLKEM768 | PASS |
| Safari 17 / iOS 17 | 回退 X25519（不支持 PQC 原生） | PASS（兼容回退） |
| Android 12+ / BouncyCastle 1.77+ | 应用层支持混合 KEM | PASS |
| VPN 客户端 v5.2（新） | 混合 IKEv2 提议 | PASS |
| VPN 客户端 v4.x（旧） | 30 天宽限期内回退 | 有条件 PASS（宽限计时） |
| TLS 1.2 遗留 Java | 无法协商混合握手 | 有条件 → 转 P3 专项工单 |

完成定义：矩阵中无 FAIL；「有条件」项必须挂接对应遗留风险登记项（见 FR-7.10）并设置到期/复检日期。

**FR-7.3（性能基准与阈值门）** 平台采集验收时的性能指标并与阈值门比对，超阈值即阻断验收并告警。基线锚定文件4实测值。

| 性能指标 | 迁移前 | 迁移后 | 增量 | 阈值门 | 判定 |
|---|---|---|---|---|---|
| TLS 握手延迟 p99 | 31.2 ms | 39.5 ms | +8.3 ms | < 15 ms 可接受 | PASS |
| 吞吐 RPS | 1840 | 1720 | -6.5% | 会话复用后目标 ~2% | 有条件 PASS（建议开启会话复用） |
| IKEv2 建立时间 | 380 ms | 412 ms | +32 ms | < 15% | PASS |
| 数据重加密速率 | — | 820 MB/s | — | 监控基线 | PASS |

完成定义：握手延迟增量 < 15 ms 且 IKEv2 建立时间增量 < 15%；吞吐降幅超目标时给出「开启会话复用」整改建议并记录为有条件项。验收口径：取 p99 而非均值，连续采样窗口 ≥ 5 分钟。

**FR-7.4（安全验证 — 降级攻击四向量）** 平台对每个混合 KEM 端点执行四向量降级/伪造攻击测试，全部必须被正确拒绝方可通过。

| 向量号 | 攻击模拟 | 期望结果（通过条件） |
|---|---|---|
| V-SEC-01 | 强制仅 X25519（剥离 PQC） | 服务器按策略拒绝（强制混合的端点） |
| V-SEC-02 | 重放 TLS 1.2 ClientHello | 被拒绝 |
| V-SEC-03 | 伪造 ServerHello | 证书验证失败，连接中止 |
| V-SEC-04 | 注入虚假 OCSP 响应 | OCSP 签名校验失败 |

完成定义：四向量全部「被拒绝/失败」即 PASS；任一未被拒绝判 FAIL 并升级为安全事件。

**FR-7.5（混合密钥材料溯源审计）** 平台审计会话密钥派生链路，确认混合性而非降级到单一算法。验收口径：确认会话密钥由 `X25519 ∥ ML-KEM-768` 的共享密钥**拼接(concatenation)后经 TLS1.3 密钥调度 HKDF-Expand-Label(HKDF-SHA-256)派生**，context/label 绑定 `hybrid-kem-v1`（对齐 draft-ietf-tls-hybrid-design）。完成定义：抓取的密钥派生证据三要素（两组件来源、HKDF 派生、context 标签）齐全且一致，任一缺失或不符判 FAIL。

**FR-7.6（验收报告一键生成与签署流）** 验收套件执行完成后，平台一键生成结构化验收报告并进入签署状态机。报告内容固定包含：测试总览（通过/有条件/未通过计数）、P1 资产覆盖率、四大类逐项结果、性能阈值比对、遗留风险登记。签署状态机如下。

| 状态 | 含义 | 触发流转 | 权限 |
|---|---|---|---|
| DRAFT | 套件执行完毕自动生成 | 提交评审 | 系统 |
| UNDER_REVIEW | 评审中 | 评审通过 / 退回 | 评审人 |
| SIGNED | 验收签署，资产置「已完成」 | 归档 | 责任人（双签可选 DP-7.A） |
| REJECTED | 退回改造阶段 | 回 ④ Remediate | 评审人 |

完成定义：按验收 Gate 锁定口径（与 1.8 KPI、7.1 一致：「通过」+「有条件通过」= 47/47 且「未通过」=0，纯「通过」≥44，有条件项须挂遗留风险登记方计入达标），仅当 0 项 FAIL 且全部「有条件」项已挂风险登记时方可流转至 SIGNED；签署动作写入审计日志（操作人、时间、报告哈希）。

> **DP-7.A（章节级，非顶层台账）验收签署是否强制双签？** 推荐：P1 资产强制「改造责任人 + 安全评审人」双签，P2/P3 单签即可，平衡治理刚性与吞吐。

### 7.2 持续监测子系统（Monitor）

验收通过仅代表迁移完成的一个时间点，持续监测保证已迁移资产长期不回退、CBOM 长期不失真、新威胁能触发复评。

**FR-7.7（周期性复扫与密码漂移/回退检测）** 平台按可配置周期（默认每季度，与摸底报告更新节奏一致）调用发现引擎复扫已纳管资产，对比上次基线，识别「密码漂移」：已迁混合端点回退到纯经典算法、新增未纳管的量子脆弱使用点、算法配置被人工改回。完成定义：每次复扫输出 diff 报告（新增/消失/降级三类变更），凡检出「混合→经典回退」立即产生 P1 告警并自动开复评工单。

**FR-7.8（SLO 监测 — 握手成功率/延迟）** 平台对运行态混合 KEM 端点持续采集 SLO 指标，按下表阈值告警（详见 FR-7.12 SLO 表）。完成定义：握手失败率连续窗口超 0.1% 触发告警；p99 延迟超验收基线绝对值 + 容差触发预警。

**FR-7.9（证书到期预警）** 平台跟踪所有纳管证书（混合根 CA、服务器证书、代码签名、IoT/嵌入式证书）有效期，分级提前预警。完成定义：根 CA / 中间 CA 到期前 180 天、服务器证书 30 天、长有效期 IoT 证书（5–10 年）到期前 365 天产生预警工单；无 OTA 能力证书（见 R-003）额外标注「需现场检修替换」。

**FR-7.10（遗留风险登记跟踪）** 平台以登记台账跟踪验收时未闭合的遗留风险，沿用工具包 R-00x 编号风格，每项含 编号 / 描述 / 等级 / 处置路径 / 状态 / 复检日期。初始台账如下。

| 编号 | 描述 | 等级 | 处置路径 | 状态 |
|---|---|---|---|---|
| R-001 | TLS 1.2 遗留 Java 无法混合握手 | 中 | P3 专项工单 #2341 | 跟踪中 |
| R-002 | TSA 供应商升级（时间戳） | 中 | 待供应商升级 | 跟踪中 |
| R-003 | IoT/工控 47 台 5–10 年证书无 OTA | 高 | 年度检修替换 | 跟踪中 |
| R-004 | iOS ML-KEM 原生支持待 Apple | 中 | 等待上游 | 跟踪中 |

完成定义：每项设复检日期，到期未处置自动提醒责任人；R-003 类「高」级风险纳入 SLO 看板常显，闭合需上传替换/升级证据。

**FR-7.11（量子威胁情报订阅与复评触发）** 平台订阅量子计算威胁情报源，跟踪 NIST FIPS 203/204/205 标准更新、国密 PQC 标准进展、量子计算能力突破（逻辑量子比特里程碑，参照威胁模型约 4000 逻辑量子比特破 RSA-2048）。完成定义：情报源更新映射为复评触发规则——出现「相关算法被攻破/标准弃用/量子能力跨越阈值」时，自动对受影响资产类型批量重算五维风险分（联动评估引擎）并生成复评批次。

**FR-7.12（CBOM 自动持续更新）** 复扫结果自动回写 CBOM（CycloneDX 格式），保持密码资产清单与现网一致，无需人工重建。完成定义：每次复扫后 CBOM 版本号自增、记录变更集（新增/移除/算法变更密码资产），并保留历史版本可 diff；与文件1-2「每季度更新」节奏对齐，做到自动化而非手工。

## 8. 系统架构与非功能需求

本章定义 zhulong-pqm 平台的逻辑/部署架构、组件职责、技术栈、多租户与 RBAC 模型、非功能需求(NFR)及开放集成 API。架构服务于第 3 章定义的「发现-建档-评估-改造-监测」五阶段闭环,所有组件均围绕该主线编排。NFR 一律给出可测量指标与验收口径;凡涉及战略分叉以章节级决策点 DP-8.x 形式标注(回链权威台账见 11.1)。

### 8.1 逻辑架构与组件职责

平台采用 Go 微服务分层架构,自下而上分为采集层、数据层、分析层、编排层、接入层。组件清单与映射的五阶段如下:

| 组件 | 代号 | 所属层 | 服务的阶段 | 核心职责 |
|---|---|---|---|---|
| 扫描引擎集群 | scan-engine | 采集层 | ①发现 | 调度内置 30 项规则库(L1-L4),分布式执行 nmap/sslyze/testssl.sh/ssh-audit/Semgrep/Syft 等探针 |
| 轻量 Agent（可选 / R2） | pqm-agent | 采集层 | ①发现 | **辅助补盲组件，非 R1 强制**（按 DP-02）：部署于隔离网段/主机侧,采集网络扫描覆盖不到的本地证书/私钥库、SSH 主机密钥、本地库版本、本地代码/SBOM,回传扫描引擎 |
| CBOM 存储服务 | cbom-store | 数据层 | ②建档 | 密码使用点清单的规范化存储、版本化(每季度更新)、CycloneDX CBOM 导入导出 |
| 评分引擎 | score-engine | 分析层 | ③评估 | 执行五维风险模型(D1-D5 加权)、分级 P1-P4、HNDL 自动标记、预设画像匹配 |
| 改造编排控制器 | remediate-ctrl | 编排层 | ④改造 | 驱动 34 任务/4 阶段项目计划、轨道编排、调用平台内建能力与外部设备下发 |
| 配置中心 | crypto-config | 编排层 | ④改造/⑤监测 | 密码敏捷(Crypto Agility)算法配置集中下发、混合 KEM 套件参数管理 |
| 监测服务 | monitor-svc | 分析层 | ⑤监测 | 持续监测 SLO(握手延迟/吞吐/失败率)、告警、CBOM 漂移检测、威胁情报订阅 |
| 验收测试服务 | verify-svc | 编排层 | ④改造 | 执行 47 项验收测试套件、兼容性矩阵、降级攻击与密钥溯源审计 |
| 控制台 | pqm-console | 接入层 | 全阶段 | Vue3+Arco 黏土橙暖色系 Web 控制台,五阶段闭环可视化、仪表板 |
| API 网关 | pqm-gateway | 接入层 | 全阶段 | 统一鉴权、限流、租户路由、开放集成 API 出入口 |

- **FR-8.1**:平台须实现上表 10 个组件并保持职责单一。**验收口径**:每个组件独立可部署、独立伸缩,组件间仅通过 API/消息队列通信,无直连数据库跨写;架构评审确认每个组件唯一映射到五阶段中的明确职责。

### 8.2 技术栈与数据流

| 层 | 技术选型 | 约束 |
|---|---|---|
| 后端 | Go 微服务 | gRPC 内部通信 + REST 对外 |
| 前端 | Vue3 + Arco Design Vue,Claude 黏土橙暖色系(accent #B4552D/#DB855C) | Arco token 须铺 `:root`/`body` |
| 主数据库 | PostgreSQL | CBOM、资产、评分、租户、RBAC、审计;支持信创国产库替换(见 8.6) |
| 消息队列 | Kafka/RocketMQ | 扫描任务分发、结果回传、漂移事件总线 |
| 对象存储 | S3 兼容/MinIO | 扫描原始证据、抓包、验收报告、CBOM 快照存证 |
| 缓存 | Redis | 评分批量计算中间态、会话、限流计数 |

数据流(文字描述,贯穿五阶段):

1. **发现→建档**:`pqm-console` 下发扫描策略 → `pqm-gateway` 鉴权路由 → 任务经消息队列分发至 `scan-engine` 集群/`pqm-agent` → 探针执行,原始证据(抓包/配置/SBOM)存入对象存储,结构化结果写入 `cbom-store`。
2. **建档→评估**:`cbom-store` 新增/变更密码使用点触发事件 → `score-engine` 拉取资产五维特征,计算综合分(`D1×30%+D2×25%+D3×20%+D4×15%+D5×10%`),回写分级 P1-P4 与 HNDL 标记到 `cbom-store`。
3. **评估→改造**:`remediate-ctrl` 按优先级生成 34 任务项目计划 → 通过 `crypto-config` 下发混合 KEM 套件参数,或经开放 API 调用外部烛龙网关/CA/HSM 执行改造。
4. **改造→监测**:`verify-svc` 执行验收测试套件并出报告 → `monitor-svc` 接管持续监测,采集 SLO 指标与 CBOM 漂移,漂移事件回灌 `cbom-store` 形成闭环。

- **FR-8.2**:平台须落地上述技术栈与四段数据流。**验收口径**:端到端贯通一次「扫描一个资产→生成 CBOM 条目→自动评分分级→下发一次混合 KEM 改造→验收测试通过→监测产生 SLO 曲线」的全链路演示;原始证据可在对象存储溯源,前端配色经检视符合黏土橙 token。

### 8.3 多租户与 RBAC 模型

平台支持私有化单租户与 SaaS 多租户两种形态(见 8.7)。租户隔离采用「共享集群 + `tenant_id` 行级隔离 + 对象存储按租户前缀分区」,关键表(CBOM、评分、审计)强制 `tenant_id` 谓词下推。

RBAC 角色与权限矩阵(C=创建/R=读/U=更新/D=删除/X=执行):

| 角色 | 资产/CBOM | 评分策略(权重/阈值) | 改造编排执行 | 配置中心下发 | 审计日志 | 租户/用户管理 |
|---|---|---|---|---|---|---|
| 平台超管 platform-admin | CRUD | CRUD | X | X | R | CRUD(跨租户) |
| 租户管理员 tenant-admin | CRUD | RU | X | X | R | CRUD(本租户) |
| 安全分析师 analyst | CRU | RU | — | — | R | — |
| 改造工程师 operator | RU | R | X | X | R | — |
| 审计员 auditor | R | R | R | R | R | R |
| 只读访客 viewer | R | R | — | — | — | — |

- **FR-8.3**:平台须实现上述 6 角色 RBAC 与 `tenant_id` 行级隔离,遵循最小权限原则。**验收口径**:跨租户越权读写返回 403 并留审计;`analyst` 无法触发改造执行;评分权重/阈值修改仅限 admin 与 analyst,且每次修改进审计;租户 A 的对象存储证据无法被租户 B 的任何角色访问。
- **DP-8.A(隔离强度)**:行级共享隔离(默认,密度高、成本低)vs. 每租户独立 Schema/库(强隔离、信创/密评合规友好)。**推荐**:SaaS 形态用行级隔离,私有化/密评严苛行业(政务/能源)用独立 Schema。

### 8.7 部署模式 NFR

**部署主干（按 DP-02）**：R1 无需强制部署主机 Agent 集群——Agentless 扫描引擎（`scan-engine`）+ 被动旁路探针即可交付完整发现能力；`pqm-agent` 为可选组件，仅在需要采集本地证书/私钥库、SSH 主机密钥、本地库版本、本地代码/SBOM 等网络扫描覆盖不到的资产时按需部署（R2 可选增量）。

| 编号 | 部署形态 | 要求 | 验收口径 |
|---|---|---|---|
| NFR-8.7.1 | 私有化 | 全栈离线部署,无外网依赖(威胁情报订阅可离线包导入) | 完全断网环境完成安装与五阶段闭环演示 |
| NFR-8.7.2 | Appliance | 一体机镜像,含国密栈与全组件,开箱即用 | 镜像导入后 ≤ 30min 完成初始化,符合烛龙家族 Appliance 交付惯例 |
| NFR-8.7.3 | SaaS 多租户 | 共享集群 + 行级隔离 + 弹性伸缩 | 多租户隔离测试通过(见 FR-8.3),扫描引擎可按队列深度自动扩缩 |

- **DP-8.B(交付优先级)**:私有化/Appliance 优先(契合烛龙信创/2B 现状)vs. SaaS 优先(规模化)。**推荐**:首版以私有化 + Appliance 为主交付形态,SaaS 作为 P2 能力。

## 9. 用户角色、权限与关键流程

本章定义 zhulong-pqm 平台的使用者角色画像、基于角色的访问控制(RBAC)矩阵、贯穿五阶段闭环(发现→建档→评估→改造→监测)的端到端用户旅程(J1–J5),以及控制台信息架构与页面清单。本章是第 3 章功能架构与第 10 章交付的桥接层:所有功能需求最终都落到"某角色在某页面执行某操作"的可验收行为上。

### 9.1 用户角色画像(Persona)

平台面向 6 类核心 Persona。每类给出主要目标、关注点、痛点,后续 RBAC 与旅程设计均围绕其展开。

| 角色 | 代号 | 主要目标 | 核心关注 | 当前痛点(工具包阶段) |
|------|------|---------|---------|------------------------|
| CISO / CIO | R-CISO | 量化全局量子风险、向董事会/监管交代迁移进度 | 风险态势、合规符合度(等保2.0/密评/CNSA 2.0)、预算与里程碑 | 手工 Excel 摸底无法滚动汇总,极高风险7项/P1高优14项缺可信看板 |
| 安全密码工程师 | R-CRYPTO | 把脆弱算法替换为 ML-KEM/ML-DSA 混合方案 | 算法选型(X25519+ML-KEM-768)、混合握手正确性、密钥溯源 | CBOM 手工维护、混合 KEM 配置易错、缺自动验收 |
| DevSecOps 工程师 | R-DEVOPS | 把改造下发到网关/反代/CA 并灰度上线 | 灰度比例(5→20→50→100%)、回退、工单与流水线 | 配置散落各设备,灰度/回退靠人盯,无统一下发面 |
| 合规风控官 | R-GRC | 证明迁移满足合规与审计要求 | 高风险资产清单、HNDL 标记、遗留风险登记(R-001..004) | 验收报告靠手工拼,审计证据链不连续 |
| 运维工程师 | R-OPS | 保障改造后系统稳定与监测告警 | 性能 SLO(握手 p99、RPS、握手失败率0.1%)、漂移告警 | 改造后性能退化无基线对照,告警无来源定位 |
| 业务系统 Owner | R-OWNER | 在不中断业务前提下完成本系统改造 | 改造窗口、兼容性(旧客户端回退)、对自身资产范围的可见性 | 不了解本系统密码资产暴露面,被动接受改造排期 |

补充:平台内置 **R-ADMIN(平台管理员)** 负责租户/用户/对接配置,与上述业务角色正交。多租户/数据范围隔离粒度作为 **DP-9.A:角色数据可见性边界**（章节级，非顶层台账）提出,推荐项=按"业务系统 Owner 仅见自己被授权的资产域,GRC/CISO 全域只读"。

### 9.2 RBAC 权限矩阵

权限动作四档:**R 读** / **W 写(创建/编辑)** / **A 审批** / **D 下发(对网关/CA/反代执行真实变更)**。模块对齐五阶段闭环 + 治理面。`—` 表示无权限。

| 模块 \ 角色 | R-CISO | R-CRYPTO | R-DEVOPS | R-GRC | R-OPS | R-OWNER | R-ADMIN |
|------------|:------:|:--------:|:--------:|:-----:|:-----:|:-------:|:-------:|
| 发现/扫描任务(Discover) | R | R/W | R/W | R | R | R(本域) | R/W |
| CBOM 资产清单(Inventory) | R | R/W | R | R | R | R(本域) | R |
| 风险评估引擎(Assess) | R | R/W | R | R | R | R(本域) | R |
| 优先级看板/批次规划 | R/A | R/W | R | R | R | R(本域) | R |
| 改造编排·混合证书签发 | R | R/W | R/W/**D** | R | R | — | R |
| 改造编排·网关混合KEM下发 | R | R/W | R/W/**D** | R | R | — | R |
| 灰度发布/工单 | R | R | R/W/**D** | R | R | R(本域)/A | R |
| 验收测试套件(Accept) | R | R/W | R/W | R/A | R | R(本域) | R |
| 监测/SLO/漂移告警(Monitor) | R | R | R | R | R/W | R(本域) | R |
| 遗留风险登记(R-xxx) | R | R/W | R | R/W/**A** | R | R(本域) | R |
| 合规/审计报告导出 | R | R | — | R/W | — | — | R |
| 平台设置/对接(HSM/CA/网关) | — | R | R | — | R | — | R/W/D |
| 用户与角色管理 | — | — | — | — | — | — | R/W |

**FR-9.1** 系统必须实现上述 RBAC 矩阵,任何涉及真实变更的操作(标记 **D**:网关混合 KEM 下发、混合证书签发、灰度推进)仅限 R-DEVOPS 与 R-ADMIN。**验收口径**:以非授权角色(如 R-OWNER)调用下发 API 返回 403 且记审计日志;矩阵每格在自动化权限测试中均有一条用例,通过率 100%。

**FR-9.2** 系统必须支持"**写-审-发**"职责分离:对网关/CA 的下发(D)动作在执行前需对应审批(A)通过;批次规划进入执行需 R-CISO 审批,遗留风险关闭需 R-GRC 审批。**验收口径**:未经审批的下发请求被拦截并提示缺失审批节点;审批人与执行人为不同账号(强制四眼原则,可在 DP-9.B 中放宽)。

**FR-9.3** 业务系统 Owner 的数据可见性必须按资产域隔离(对应 DP-9.A)。**验收口径**:R-OWNER 登录后,CBOM/评估/看板仅返回其被授权资产域内记录,跨域访问返回空集且不泄露存在性。

**FR-9.4** 所有 A/D 级操作必须留痕:记录操作者、时间、对象(资产/网关 ID)、前后值、关联工单/审批单号。**验收口径**:审计日志可按对象反查完整变更链,导出后与合规报告(9.x 旅程 J4)证据字段一一对应。

### 9.3 关键端到端旅程(J1–J5)

旅程以五阶段闭环为骨架,每步标注 **涉及模块** 与 **主责角色**。旅程编号 J1–J5 与第 3 章五阶段一一对应。

#### J1 首次摸底(Discover → Inventory)

| 步 | 动作 | 涉及模块 | 主责 |
|----|------|---------|------|
| 1 | 创建摸底范围(TLS/VPN/PKI-KMS/应用层/数据存储/供应链 6 域) | 发现/扫描 | R-CRYPTO |
| 2 | 配置并运行扫描(testssl/sslyze/nmap/Semgrep/Syft + 内置30项规则库) | 发现/扫描 | R-DEVOPS |
| 3 | 扫描结果归一化为密码使用点,自动标注当前算法/量子风险等级 | CBOM 资产清单 | 系统 |
| 4 | 生成 CycloneDX CBOM 与摸底报告(算法分布总览+分域结果) | CBOM/报告 | R-CRYPTO |
| 5 | 摸底报告呈递 CISO,锁定"总检查项30/P1高优14/极高风险7"基线 | 报告/概览看板 | R-CISO |

**FR-9.5** J1 必须在一次任务内完成"建范围→扫描→归一→生成 CBOM+报告"全链路。**验收口径**:对一个含 TLS/VPN/PKI 三域的测试环境,系统输出有效 CycloneDX CBOM(可被第三方 CBOM 工具解析)、摸底报告含算法分布总览表,且 30 项规则库覆盖率可核查(命中/未命中/不适用三态)。

#### J2 风险评估到批次规划(Assess)

| 步 | 动作 | 涉及模块 | 主责 |
|----|------|---------|------|
| 1 | 对 CBOM 全量资产运行五维评分(D1×30%+D2×25%+D3×20%+D4×15%+D5×10%) | 风险评估引擎 | 系统 |
| 2 | 自动定级 P1≥75/P2 50-74/P3 25-49/P4<25,并按规则标记 HNDL(D2≥60 且 D3≥60) | 风险评估引擎 | 系统 |
| 3 | R-CRYPTO 复核五维向量与画像(如内部根CA 86/SSL VPN 82),可人工修正并留痕 | 风险评估引擎 | R-CRYPTO |
| 4 | 优先级看板呈现(按 P1/P2/P3 分桶、极高/高/中色阶) | 优先级看板 | R-CISO |
| 5 | 一键生成迁移项目计划(34任务/4阶段/18月,套行业×规模拉伸因子) | 批次规划/计划引擎 | R-CRYPTO |
| 6 | R-CISO 审批批次进入执行 | 批次规划 | R-CISO |

**FR-9.6** 五维评分必须可解释、可复核、可修正。**验收口径**:任一资产可展开显示 D1–D5 取值、权重与加权得分;复核修正后得分实时重算并落审计;7 个内置预设画像(根CA 86/SSL VPN 82/对外TLS 41/长期档案 75/IoT证书 75/数据库 52/代码签名 74)的计算结果与共享上下文一致,误差为 0。

**FR-9.7** 生成的迁移计划必须随行业/规模因子自适应。**验收口径**:同一资产集在"能源工控×1.8"与"科技×1.0"下,工期与里程碑节点按因子线性拉伸并可在甘特视图核对;计划含 MS1–MS4 里程碑与前置依赖。

#### J3 改造执行(Remediate)

| 步 | 动作 | 涉及模块 | 主责 |
|----|------|---------|------|
| 1 | 选定 P1 资产轨道(根CA/外部TLS/SSL VPN/长期数据/代码签名) | 改造编排 | R-DEVOPS |
| 2 | 内建能力:PQC就绪CA签发混合根/中间证书(ECDSA+ML-DSA-65) | 混合证书签发 | R-DEVOPS |
| 3 | 编排外部设备:向烛龙 IPSEC/SSL VPN 网关下发 ke1_mlkem 混合提议 | 网关混合KEM下发 | R-DEVOPS |
| 4 | 反代/Nginx+liboqs 注入 X25519+ML-KEM-768,按 5→20→50→100% 灰度 | 灰度发布 | R-DEVOPS |
| 5 | 每步生成工单,异常自动回退至上一灰度档,客户端升级率门限>95% | 工单/灰度 | R-DEVOPS |

**FR-9.8** 下发动作必须经"写-审-发"(FR-9.2)且具备一键回退。**验收口径**:网关混合 KEM 下发后,`swanctl --list-sas` 可见 `ke-method:MLKEM_768+X25519`;灰度推进每档落工单;触发回退后配置在 T 秒内恢复上一档且告警归档。

**FR-9.9** 平台内建能力与外部编排须在同一编排视图统一管理。**验收口径**:一个改造批次可混合包含"签发混合证书(内建)"与"网关下发(外部)"任务,二者共享同一进度/状态机与回退入口。

#### J4 验收(Accept)

| 步 | 动作 | 涉及模块 | 主责 |
|----|------|---------|------|
| 1 | 对改造资产运行验收测试套件(协议层/兼容性/性能/安全 四类) | 验收测试套件 | R-CRYPTO |
| 2 | 协议层断言:TLS `Server Temp Key:X25519MLKEM768`、IKEv2 混合 SA、ML-DSA 双签名 | 验收套件 | 系统 |
| 3 | 性能基准对照 SLO(握手 p99 +8.3ms<15ms、RPS −6.5%、失败率0.1%) | 验收/SLO | R-OPS |
| 4 | 安全验证:降级/重放/伪造/虚假OCSP 全部被拒,密钥溯源 HKDF-SHA-256(TLS1.3 密钥调度) | 验收套件 | R-CRYPTO |
| 5 | 生成验收报告(47项 44通过/3有条件)、登记遗留风险 R-001..004 | 验收报告/风险登记 | R-GRC |
| 6 | R-GRC 复核、CISO 签署,关闭批次或转工单 | 验收报告 | R-CISO/R-GRC |

**FR-9.10** 验收套件必须可一键执行并生成可签署报告。**验收口径**:对 P1 资产输出测试项明细(每项 通过/有条件/未通过 三态)、性能基准与 SLO 阈值对照、安全验证结论;有条件项自动转遗留风险登记(如 R-001 TLS1.2 遗留 Java→P3 工单)。

**FR-9.11** 验收报告须形成可审计签署链。**验收口径**:R-GRC 复核 + R-CISO 电子签署后报告锁定为只读版本,签署元数据(签署人/时间/报告哈希)进入审计日志,可与 FR-9.4 变更链交叉核验。

| 步 | 动作 | 涉及模块 | 主责 |
|----|------|---------|------|
| 1 | 周期性复扫(CBOM 持续自动更新)+ 订阅量子威胁情报 | 监测/发现 | 系统 |
| 2 | 检测密码学漂移(新增脆弱点/证书过期/算法回退)并告警 | 漂移告警 | 系统 |

**FR-9.12** 监测必须构成闭环:漂移/告警可自动回灌评估并生成处置项。**验收口径**:注入一个模拟漂移(如算法回退为纯 X25519),系统在监测面产生告警、在 CBOM 标记变更、并自动创建一条复评/工单,形成 J5→J2 闭环可追踪。

| 验收 | 测试套件 / 验收报告 | 运行四类测试、生成与签署报告 | R-CRYPTO/R-GRC |

| 治理 | 遗留风险登记 / 合规报告 / 审计日志 | R-xxx 风险池、合规导出、A/D 操作留痕 | R-GRC |
| 设置 | 对接管理 / 用户与角色 / 租户 | HSM/CA/网关对接、RBAC 配置、多租户 | R-ADMIN |

**FR-9.13** 控制台信息架构必须与五阶段闭环逐项对应,且每个页面的可见性/可操作性受 9.2 RBAC 约束。**验收口径**:不同角色登录后菜单与页内操作按钮按矩阵动态裁剪(如 R-OWNER 无"证书签发""下发"按钮);每个一级菜单至少有一条 J1–J5 旅程步骤落入,无悬空页面。

**FR-9.14** 概览驾驶舱必须聚合五阶段关键指标供 CISO 一屏决策。**验收口径**:驾驶舱实时呈现总检查项 30、P1 高优 14（= 极高风险 7 项 + 高优先级 7 项，完整清单见规则库导出）、极高风险 7、当前迁移进度(对照 MS1–MS4)、合规符合度及最近告警,数据来源可下钻至对应模块页面。

## 10. 产品路线图与版本规划

本章定义烛龙·后量子迁移治理平台(zhulong-pqm)从「手工模板工具包」演进为「自动化迁移治理平台」的版本路线图。路线图以第 1 章五阶段闭环(发现→建档→评估→改造→监测)为骨架，按「先打通价值闭环、再做规模化编排、最后做持续治理与生态」分三个主版本(R1/R2/R3)交付，并以四份工具包模板(脆弱性发现清单、CBOM 摸底报告、五维评分模型、迁移项目计划+验收报告)作为各版本能力的成熟度基线。

### 10.1 版本节奏与演进总则

- 主线原则：每个版本必须独立交付端到端价值，禁止「半截闭环」。R1 必须能让一个客户从扫描走到拿到 CBOM 与评分报告并完成至少一类资产的实际改造下发；R2/R3 在此之上做横向铺开与纵向加深。
- 成熟度阶梯：以工具包模板的「手工 vs 自动化」差距定义演进，每个版本明确把哪几份模板从「人填表」升级为「平台产出」。
- 版本间不破坏数据：R1 建立的 CBOM/评分数据模型为后续版本的稳定基线，R2/R3 只做扩展不做不兼容重构(数据契约见第 7 章)。
- 行业/规模自适应因子(金融×1.3、政务×1.0、医疗×1.4、能源工控×1.8、科技×1.0；大型×1.3/中型×1.0/小型×0.7)在 R2 项目计划引擎中落地，R1 暂用固定 18 个月模板。

### 10.3 R1(MVP) — 价值闭环打通

定位：把工具包文件 1-1/1-2/2-1 三份模板产品化，先打通「发现→CBOM→评分→报告」数据闭环；发现侧仅交付 Agentless（被动流量可选，DP-02），改造侧以**编排外部设备为主线（DP-03）**——只做最高确定性、最高 ROI 的一条龙（**编排烛龙网关混合 KEM 下发 + HSM 混合根 CA**），内建 PQC 就绪 CA/混合反代后置 R2。

- FR-10.1：发现引擎内置工具包文件 1-1 的 L1(网络协议 8 项)+ L2(证书密钥 8 项)共 16 项检查规则，**仅交付 Agentless 网络扫描（被动流量探针为可选），不含主机 Agent**（按 DP-02），对接 nmap/sslyze/testssl.sh/ssh-audit/ike-scan/openssl 等扫描器输出。验收口径：对一个含 TLS 服务、VPN 网关、内部 CA 的样板环境一键扫描（无需部署主机 Agent），能识别出 ≥14/16 项并落库为密码使用点，且自动标出 7 个极高风险项中落在 L1/L2 的项。
- FR-10.2：CBOM 引擎按 CycloneDX 格式自动生成密码资产清单，并产出摸底报告(执行摘要+算法分布总览表+分域结果+高风险资产清单+迁移建议)。验收口径：导出的 CBOM 通过 CycloneDX 1.6 schema 校验；报告含算法选型建议表(如 ECDH/RSA-OAEP→X25519+ML-KEM-768)。
- FR-10.3：五维评分引擎实现 D1×30%+D2×25%+D3×20%+D4×15%+D5×10% 综合评分，输出 P1≥75/P2=50-74/P3=25-49/P4<25 分级，并在 D2≥60 且 D3≥60 时自动打 HNDL 标记。验收口径：录入 7 个预设画像，平台算分须与基线一致(内部根 CA=86、SSL VPN 网关=82、对外 TLS=41、长期合规档案=75、IoT 证书=75、数据库静态加密=52、代码签名=74)，分级与极高/高/中标注一致。
- FR-10.4：改造编排以**编排外部设备为主线（DP-03）**：R1 打通「编排烛龙 IPSEC/SSL VPN 国密网关下发 ke1_mlkem 混合 IKEv2 提议 + 编排 HSM/加密机在硬件内生成并托管混合根 CA 密钥对（私钥不出 HSM）」一条龙；轨道①根 CA 信任链由 HSM 内生成密钥、第三方/外部 CA 签发承载，**平台内建 PQC 就绪 CA 签发与混合 KEM 反代（Nginx+liboqs 灰度）作为兜底能力后置 R2**。验收口径：在样板环境完成一台烛龙网关混合 KEM 配置下发（`swanctl --list-sas` 显示 `ke-method: MLKEM_768+X25519`）+ 一对 HSM 内生成的混合根 CA 密钥（PKCS#11 句柄可查、私钥不出 HSM），且对接入端 `openssl s_client -groups X25519MLKEM768` 能观测到 `Server Temp Key: X25519MLKEM768`。
- FR-10.5：交付形态为单租户私有化 Appliance，前端 Vue3+Arco「Claude 黏土橙」暖色系，后端 Go 微服务。验收口径：可离线安装部署，无需公网依赖完成 FR-10.1~10.4 全流程。

R1 退出标准(Definition of Done)：
1. 至少 1 个真实/样板客户环境完成「发现(仅 Agentless)→CBOM→评分→报告」全链路，并经**编排烛龙网关混合 KEM 下发 + HSM 混合根 CA**对 ≥1 类资产完成实际改造下发与握手验证；
2. 7 个预设画像评分回归 100% 通过；CBOM 通过 schema 校验；
3. 改造下发的混合 KEM 握手可被 `openssl s_client`/网关侧(`swanctl --list-sas`)实测确认，HSM 内混合根 CA 私钥不出设备；
4. 全流程可在私有化离线环境、无需部署主机 Agent 跑通。

### 10.4 R2 — 全轨道改造编排 + 验收自动化 + 计划引擎

定位：把工具包文件 3-1/3-2(项目计划)与文件 4(验收报告)产品化，发现规则补齐到 30 项并**新增主机 Agent 辅助补盲（DP-02）**，改造从一条龙扩到 5 条轨道并**补齐内建 PQC 就绪 CA/混合 KEM 反代作为兜底能力（DP-03）**，引入「计划—执行—验收」工程化能力。

- FR-10.6：发现引擎补齐 L3(应用数据 8 项)+ L4(供应链 6 项)，达成工具包文件 1-1 全部 30 项规则，对接 Semgrep/Snyk/Syft(SBOM)，并**新增主机 Agent 作为辅助补盲（DP-02 可选增量）**，覆盖网络扫描触达不到的本地证书/私钥库、SSH 主机密钥、本地库版本、本地代码/SBOM。验收口径：进度仪表板能正确统计总检查项 30、P1 高优 14、极高风险 7；在一台主机部署 Agent 后能补采本地私钥库/库版本并归一入 CBOM。
- FR-10.7：改造编排补齐轨道③SSL VPN 混合 KEM(固件验证→混合 IKEv2 提议→客户端升级>95%)、轨道④长期数据重加密(HNDL 数据集识别→新封装方案→批量重加密)、轨道⑤代码签名(RSA+ML-DSA 双签名→TSA 时间戳升级)，扩展外部设备编排(第三方 CA、OQS-OpenSSL；HSM 一条龙已在 R1 落地)，并**补齐平台内建 PQC 就绪 CA/混合 KEM 反代作为编排兜底能力（DP-03，外部设备无法覆盖时启用）**。验收口径：5 条轨道均可在平台内创建任务并驱动至少 1 个资产改造闭环；内建 PQC 就绪 CA 能签发一张 `openssl verify` 通过的混合证书链作兜底。
- FR-10.8：项目计划引擎实现工具包文件 3-1 的 34 任务/4 阶段/18 月甘特图(每格半月)、里程碑 MS1-MS4、前置依赖与完成%，并叠加行业(金融×1.3/政务×1.0/医疗×1.4/能源工控×1.8/科技×1.0)与规模(大型×1.3/中型×1.0/小型×0.7)拉伸因子自适应周期。验收口径：给定行业+规模，引擎自动生成拉伸后的甘特图与里程碑日期，与人工套用工具包模板结果一致。
- FR-10.9：验收自动化套件实现工具包文件 4 的协议层验证(TLS/IKEv2 混合 KEM 握手、回退、证书链、OCSP、ML-DSA 双签名)、兼容性矩阵、性能基准与安全验证(降级/重放/伪造/虚假 OCSP)，并自动登记遗留风险(R-001~R-004)。验收口径：对 R1/R2 改造资产跑一轮，能复现「47 项 / 44 通过 / 3 有条件 / 0 未通过」量级的结构化验收报告，性能基准比对 SLO 阈值(握手延迟 +8.3ms < 15ms 可接受、吞吐 -6.5%)。

R2 退出标准：
1. 发现 30 项规则全开且仪表板指标正确(30/14/7)；
2. 5 条改造轨道全部具备至少一个端到端样例；
3. 计划引擎对 ≥3 个行业×规模组合输出正确拉伸结果；
4. 验收套件能产出含遗留风险登记表的完整验收报告，并自动判定 SLO 是否达标。

### 10.5 R3 — 持续监测 + 威胁情报 + 多租户/信创 + 开放生态

定位：把闭环最后一环「监测」常态化，并完成从「项目交付」到「平台托管/生态集成」的跃迁，对齐 IBM Quantum Safe Remediator、Venafi/Keyfactor 等竞品的持续治理能力。

- FR-10.10：持续监测实现全量 SLO 看板与告警(TLS 握手延迟 p99、吞吐 RPS、IKEv2 建立时延、数据重加密速率)，握手失败率告警阈值 0.1%，并对遗留风险(如 R-003 IoT 工控 47 台无 OTA)做工单化跟踪与年度复检提醒。验收口径：监测面板可实时拉取改造资产的握手成功率，失败率越阈即告警并生成工单。
- FR-10.11：CBOM 持续自动更新与评分动态刷新——监测发现新密码使用点或算法变化时，自动回写 CBOM 并触发重新评分(对接工具包「每季度更新」节奏)。验收口径：模拟新增一个量子脆弱资产，平台在下一周期自动纳入 CBOM 并产出更新后的评分与差异报告。
- FR-10.12：威胁情报订阅(量子威胁情报 + 算法弃用预警)与密码敏捷配置中心(Crypto Agility，算法配置中心化、一处变更多处下发)。验收口径：可订阅情报源并在算法被标记弃用时主动预警关联资产；配置中心改一次算法策略可下发到多个改造点。
- FR-10.13：平台底座升级为多租户 + 信创/Appliance 适配 + 开放 API/生态。验收口径：单实例支持多租户隔离(数据/权限);通过信创环境(国产 CPU/OS)部署验证；提供 OpenAPI 供第三方系统(SOC/CMDB/CMP)拉取 CBOM、评分、迁移进度。

R3 退出标准：
1. 监测闭环上线，失败率阈值 0.1% 告警与工单跟踪可用；
2. CBOM 与评分形成「监测→回写→重评分」自动循环；
3. 多租户隔离 + 信创部署 + 开放 API 三项均通过验收；
4. 五阶段闭环全部由平台自动化驱动，工具包四份手工模板 100% 被平台能力替代。

下列为各 DP 的展开（背景/选项/推荐/影响）：

- **DP-01 产品命名与烛龙归属**（已决策 2026-06-24）。背景：当前仅为 4 份手工模板工具包，需确立产品身份。选项：(a) 独立新品牌；(b) 纳入烛龙家族命名为 Zhulong PQM。已采纳：(b)，正式命名「烛龙·后量子迁移治理平台 / Zhulong PQM」，统一品牌归入烛龙国密家族。影响：沿用 Claude 黏土橙视觉与国密/Appliance 交付通道；控制台独立自建（DP-10 已决策为独立控制台，不与现有烛龙控制台共用入口）。
- **DP-02 Agent vs Agentless**（已决策 2026-06-24）。背景：30 项发现规则中，L1 网络/L2 证书多可远程探测，L3 代码（Semgrep）、L2 SSH 主机密钥、L4 SBOM(Syft) 需主机侧采集。选项：(a) 纯 Agentless；(b) 纯 Agent；(c) Agentless 默认+Agent 可选。已采纳：(c) 的明确化——**Agentless 网络扫描为主干、强制能力，被动流量分析为其增强手段；主机 Agent 仅作辅助补盲**，覆盖网络扫描天然触达不到的本地资产（主机证书/私钥库、SSH 主机密钥、本地密码库版本、本地代码/SBOM）。影响：R1 仅交付 Agentless（含被动流量为可选），主机 Agent 列为 R2 可选增量（关联第 3 章发现引擎）。
- **DP-03 内建 vs 编排**（已决策 2026-06-24）。背景：改造手段两类并存。选项：(a) 重投入自研全栈改造能力；(b) 以编排外部设备为主、内建仅兜底。已采纳：(b)——**改造主线 = 编排外部设备**（烛龙 IPSEC/SSL VPN 国密网关下发 ke1_mlkem 混合 IKEv2 提议、HSM/加密机生成与托管混合根 CA 密钥、第三方 CA 签发混合/ML-DSA 证书）；内建密码能力（PQC 就绪 CA、混合 KEM 反代、密码敏捷配置中心）重定位为"编排的补充与兜底"，**不追求自建完整密码基础设施**。影响：R1 改造先打通"编排烛龙网关混合 KEM 下发 + HSM 混合根 CA"一条龙，内建 CA/反代后置到 R2；避免与 HSM/CA/VPN 厂商正面造轮子，聚焦治理与编排价值。
- **DP-04 国密 PQC 等待策略**（已决策 2026-06-24）。背景：GM/T 国密 PQC 标准尚未发布，但 HNDL 威胁不等人。选项：(a) 等标准落地再动；(b) 先混合预案 SM2+ML-KEM / SM2+ML-DSA。已采纳：(b)——正式采 SM2+ML-KEM（密钥封装）/ SM2+ML-DSA（签名）混合作为已承诺的过渡方案。影响：算法选型建议表中"国密融合"列过渡项锁为 SM2+ML-KEM/SM2+ML-DSA、长期项标注"待国密 PQC 标准对齐"，平台须支持算法版本平滑切换（关联 DP-11 与密码敏捷配置中心）。
- **DP-05 多租户/SaaS**。背景：用户主线为私有化/信创/Appliance。选项：(a) V1 即多租户 SaaS；(b) V1 单租户、V2 预留多租户。推荐：(b)。影响：数据驻留无歧义（关联 DP-09），架构层预留租户隔离接口以降低 V2 改造成本。
- **DP-06 信创适配优先级**。背景：交付面向信创/政务。选项：(a) 信创延后；(b) V1 纳入信创基线。推荐：(b)。影响：发现引擎/CA/反代须在麒麟/统信 + 鲲鹏/飞腾上验证，依赖 GM 库与 GM strongSwan。
- **DP-07 CBOM 标准选型**。背景：建档需统一格式。选项：(a) 自定义 schema；(b) CycloneDX CBOM。推荐：(b)。影响：与竞品(IBM/InfoSec Global)及供应链 SBOM(Syft) 生态互通，避免锁定（关联第 5 章建档/CBOM）。
- **DP-08 被动流量分析是否纳入 R1**（已决策 2026-06-24）。背景：被动旁路分析能补主动扫描盲区，但工程复杂。选项：(a) R1 纳入为可选增强；(b) R1 仅主动扫描+配置审计+SBOM，R2 再加被动分析。已采纳：(a)——被动旁路探针**纳入 R1 作为可选增强**（默认关闭，Agentless 主动扫描仍为强制主干），与第 3 章 M2「R1 可选」表述一致。影响：R1 范围仍以主动扫描为确定交付物，被动探针作为可选开关按需启用（关联 DP-02）。
- **DP-09 数据驻留与证据保存期**。背景：扫描结果含敏感拓扑与脆弱点，属高敏数据。选项：(a) 允许云端聚合；(b) 全私有化驻留+分级保存。推荐：(b)。影响：CBOM/扫描证据 ≥3 年、验收/审计 ≥10 年，呼应"长期合规档案 10 年+"画像与密评要求。
- **DP-10 与烛龙控制台合并入口**（已决策 2026-06-24）。背景：烛龙控制台已承载统一登录/RBAC。选项：(a) 完全独立控制台；(b) 复用控制台单点登录、PQM 作子应用。已采纳：(a)——PQM 为**独立控制台**：自建登录/RBAC/审计与门户，数据面独立；仅在品牌与视觉（Claude 黏土橙）上对齐烛龙家族，不与现有烛龙控制台共用入口。影响：交付解耦、可独立私有化部署、身份体系自洽；代价是与烛龙控制台需各自维护身份/审计（后续如需统一可在 V2 评估 SSO 对接）（关联 DP-01）。
- **DP-11 评分权重是否允许自定义**。背景：D1×30%+D2×25%+D3×20%+D4×15%+D5×10% 为默认权重。选项：(a) 硬锁；(b) 默认锁定+专家模式可调权。推荐：(b)。影响：客户可按行业偏好调权，但每次调权须留痕并触发全量复算与版本对比，保证可解释（关联第 5 章评估引擎）。
- **DP-12 HNDL 时间窗假设**。背景：威胁背景设定 Shor 算法 + 约 4000 逻辑量子比特。选项：(a) 设定固定 Q-Day 倒计时；(b) 不赌具体年份，按"今日截获→未来解密"恒定威胁建模。推荐：(b)。影响：HNDL 自动标记规则（D2≥60 且 D3≥60）独立于年份假设，数据生命周期 D3 维度成为时间窗的代理变量。

### 11.5 术语表

| 术语 | 全称/含义 |
|------|-----------|
| PQC | 后量子密码(Post-Quantum Cryptography)，抗量子计算攻击的密码算法族 |
| ML-KEM | Module-Lattice KEM(FIPS 203)，格基密钥封装机制，本平台用 ML-KEM-768 |
| ML-DSA | Module-Lattice DSA(FIPS 204)，格基数字签名，本平台用 ML-DSA-65 |
| SLH-DSA | Stateless Hash-based DSA(FIPS 205)，无状态哈希签名，长期/高保障备选 |
| HNDL | Harvest Now, Decrypt Later，先截获密文、待量子机成熟后再解密的威胁模型 |
| Hybrid KEM | 混合密钥封装，经典+PQC 并联(如 `X25519 ∥ ML-KEM-768` 的共享密钥拼接(concatenation)后经 TLS1.3 密钥调度 HKDF-Expand-Label(HKDF-SHA-256)派生，context/label 绑定 `hybrid-kem-v1`，对齐 draft-ietf-tls-hybrid-design) |
| CBOM | Cryptography Bill of Materials，密码物料清单，本平台采用 CycloneDX CBOM |
| QVA | Quantum-Vulnerable Algorithm，量子脆弱算法(如 RSA/经典 DH/ECC/SM2) |
| Crypto Agility | 密码敏捷性，算法可配置、可版本化、可平滑切换的工程能力 |
| HSM | Hardware Security Module，硬件安全模块/加密机，长期密钥材料载体 |
| Shor / Q-Day | Shor 算法可破解经典公钥的量子能力门槛(约 4000 逻辑量子比特)；Q-Day 为其成真之日，本平台不预设具体年份 |
