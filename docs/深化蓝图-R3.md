## ⑤ 持续监测（Monitor）—— 补齐闭环第五环

### 目标与价值

补齐 PRD 第 7.2/7.3 节定义的「监测治理 M-MON」整环——平台当前 R1/R2 止于「改造编排」，闭环在「改造完成」处断裂，已迁资产是否回退、证书是否临期、SLO 是否越界、CRQC 进展是否触发复评全部无人值守。本领域把验收时建立的基线转为**长期 SLO 守护 + 密码漂移检测 + 证书到期预警 + 量子威胁情报订阅 + CBOM 快照 diff**，并以「复评信号（ReassessTrigger）」自动回灌 ③ 评估，让闭环从一次性项目变为可循环治理。现在做的原因：R2 已产出可被监测的改造工单与设备遥测来源，监测是把「改造→已验收→监测中→复评」状态机（PRD 2.6）真正跑通的最后一块拼图。

### 后端 API

新增 `internal/api/monitor.go`，全部挂 `/api/v1` JWT 组。写操作（PUT/POST 处置类）建议限 `admin`/`operator`（与 PRD 9.x RBAC 中 R-OPS 对监测 R/W 一致；当前代码 role 已在 ctx，按现状先放开读、写校验 role≠viewer）。

**监测策略**
- `GET /monitor/policies` → `[]MonitorPolicy`（含计算出的 `nextRunAt`）
- `POST /monitor/policies` → 创建策略，校验阈值合理性
- `PUT /monitor/policies/:id` → 全量更新（保留主键/时间）
- `DELETE /monitor/policies/:id`
- `POST /monitor/policies/:id/run` → 立即触发一次复扫（异步 goroutine，复用 Monitor Runner，对齐 `createScan` 的 `go runner.Run(context.Background(), id)` 模式）；返回 202 + 本次生成的 `MonitorEvent` 摘要

**SLO 时序**
- `GET /monitor/slo/summary` → 每个 SLO 编号最新值 + 阈值 + 状态（红/橙/黄/正常），SLO-05/06/07 由事件与资产聚合补齐，响应即仪表板 SLO 卡片数据源
- `GET /monitor/slo/series?code=SLO-02&assetId=&from=&to=` → 时序点数组，供趋势图
- `POST /monitor/slo/ingest` → 遥测回填端点（机机接口，对齐 PRD M-API「回收遥测」）。请求体 `{sloCode, assetId, remediationId, value}`；落 `SLOMetric`，越阈即生成 `MonitorEvent`。**演示态**：无真实网关遥测时由复扫/定时器合成基线附近抖动值，诚实标注 `source:"synthetic"`（沿用代码库「离线诚实模拟」风格）

**告警台账**
- `GET /monitor/events?kind=&severity=&status=` → 分页/过滤列表
- `POST /monitor/events/:id/ack` → 处置（写 `AckedBy`=ctx username，状态→acked）
- `POST /monitor/events/:id/resolve` → 闭合（写 `ResolvedAt`）
- `POST /monitor/events/:id/reassess` → 一键生成复评批次：将关联资产状态置回 `discovered`→ 触发重评分（调用 `scoring.Score`），回填 `ReassessTaskID`，形成 J5→J2 闭环（FR-9.12）

**遗留风险登记**
- `GET /monitor/legacy-risks` → R-00x 台账（支持按 level/status/recheckDate 筛选）
- `POST /monitor/legacy-risks` / `PUT /monitor/legacy-risks/:id` / `POST /monitor/legacy-risks/:id/close`（close 要求 `evidenceUrl` 非空，FR-7.10 完成定义）

**威胁情报**
- `GET /monitor/intel` → 情报条目列表
- `POST /monitor/intel` → 手工录入/拉取入库；若 `triggerReassess=true`，对 `affectedAlgos` 命中的资产批量重评分并生成复评批次（FR-7.11）
- `POST /monitor/intel/pull` → 定时/手动拉取（离线包导入或预置源），幂等去重

**CBOM 快照与 diff**
- `GET /monitor/snapshots` → 快照列表
- `POST /monitor/snapshots` → 冻结当前 CBOM 为命名快照（复用 `cbom` 包导出）
- `GET /monitor/snapshots/diff?base=:id&target=:id` → 结构化 diff（新增/移除/算法变更/证书续期/状态变化/风险等级变化 + 算法分布环比，FR-4.6.2/7.12）

**仪表板聚合**
- `GET /monitor/dashboard` → 一次性返回 SLO summary、按严重度计数的告警、证书到期甘特数据（90 天内临期证书）、漂移趋势、复评队列、R-00x 常显项（FR-7.13）

响应/错误体沿用现有 `gin.H{"error": ...}` 约定；离线/无遥测返回 200 + 合成数据，绝不 500（对齐 `testDevice` 风格）。

### 编排/算法/计算逻辑

#### 定时调度机制（`internal/monitor` 包）

仿 `scan.Runner`/`remediate.Orchestrator`：核心是同步 `Run(ctx, policyID uint)`，由调度器或 API 放入 goroutine。新增**进程内 Scheduler**：

```
monitor.Scheduler
  - 单个 time.Ticker（60s tick，对齐 NFR-8.4.5 监测时延 ≤30s、FR-6.1 健康探测 ≤60s）
  - 每 tick：扫描所有 Enabled 策略，cron 到点 → go runner.Run(ctx, policyID)
  - cron 解析用 robfig/cron 的 cron.ParseStandard（或最小自实现季度判断），算 NextRunAt
  - 由 main.go 在 srv.Router() 前启动：monitor.NewScheduler(db).Start(ctx)
```

选用进程内 ticker（非外部 cron）原因：私有化离线部署（NFR-8.7.1），无外部依赖，与 SQLite 单进程形态一致。`stepPause` 风格的可观测性保留——每次复扫落多条事件供前端轮询。

#### 复扫 → 漂移检测（FR-7.7，SLO-05）

`monitor.Runner.Run` 流程：
1. 按策略 Scope 取纳管资产（`Status in (accepted, monitored)` 优先，演示态退化为全量）
2. 对每个资产调用发现引擎复扫（复用 `scan` 包对 TLS 端点的探测）得新算法指纹
3. 与资产 `Algorithm`/`SuggestedAlgo` 对照，**漂移判定**：
   - **混合→经典回退**（如已迁 `X25519MLKEM768` 复扫得纯 `X25519`/`ECDH`）→ `Severity=p1`，立即开复评工单（SLO-05 目标 0，检出即 P1 + 复评，PRD 行 1040）
   - **新增未纳管脆弱使用点** → `Severity=warning`
   - **证书续期/更换** → `cbom_diff` 事件
4. 复扫结果自动回写 CBOM 并自增快照版本（FR-7.12）

#### 证书到期分级预警（FR-7.9，SLO-06）

每 tick 扫描 `CryptoAsset.CertNotAfter`（R1 已采集），按资产层级/类型分级提前量比对：

```
daysLeft = CertNotAfter - now
warn 条件（命中即生成 cert_expiry 事件，severity 据剩余天数）：
  根/中间 CA（Layer=L4 或 name 含 CA）:        daysLeft ≤ 180
  服务器证书（Layer=L1/L2，exposure≠internal）: daysLeft ≤ 30
  IoT/长效证书（Layer=L1 且 system 含物联网/IoT）: daysLeft ≤ 365
无 OTA 证书（关联 R-003）额外标注「需现场检修替换」
```

#### SLO 越界判定（FR-7.8，SLO-01~04）

连续窗口判定（PRD「连续窗口 ≥0.1%」「持续超」）：对每个 `(SLOCode, AssetID)`，取最近 N 个 `SLOMetric`（默认窗口 3 点），全部越界才置 `Breached` 并生成 `slo_breach` 事件，避免单点抖动误报。阈值取策略快照值。

#### 威胁情报 → 复评映射（FR-7.11）

情报条目 `Category in (algo_break, algo_deprecate)` 或 `QubitCount ≥ 4000`（破 RSA-2048 阈值）→ 对 `AffectedAlgos` 命中的资产**批量重算五维**（调 `scoring.Score`，通常抬升 D1），生成复评批次，置资产回 `已评分` 态（PRD 2.6 复评是唯一回灌入口）。

#### CBOM 快照 diff（FR-4.6.2/7.12）

按资产 `CertFingerprint`/`Endpoint` 指纹两两比对两快照：分类 `added`/`removed`/`algo_changed`/`cert_renewed`/`status_changed`/`risk_changed`；`AlgoDistJSON` 求环比（如 RSA 占比 30%→22% 标「迁移进展」）。

## ④ 验收自动化 —— 47 项测试套件 + 验收报告

### 目标与价值
补的是闭环第 ④ 环（改造）的出口与第 ⑤ 环（监测）的入口：把工具包文件4「PQC 迁移验收报告（47 项实测）」从一次性人工跑命令+留截图，产品化为**声明式、可重复、可回归、可签署**的验收套件。现在做的理由：R2 改造编排已把工单跑到 `done`，但 PRD 状态机硬约束「已验收必须满足验收套件回灌」（2.5 边界原则③、状态机约束②）——没有这一环，任何改造工单都无法合法流转到「已验收/监测中」，闭环断裂。本领域提供「改造 done → 一键跑验收 → 逐项 pass/有条件/fail → 一键生成可签署报告」的关键接缝。

### 编排/算法/计算逻辑

**Runner 状态机**（照抄 `scan.Runner`/`Orchestrator` 模式：同步 Run 由 goroutine 调，每用例落库后 `stepPause` 停顿供前端轮询）：
`pending → running →（逐用例判定）→ done/failed`。

**用例集选择**：按 `Run.Track` 从静态 `cases` 过滤 `Tracks` 命中项；轨道无关的通用项（兼容性矩阵、安全四向量、密钥溯源）全轨道纳入。基线总数锚定 **47**（V-PROTO ×6 + V-COMPAT ×6 + V-PERF ×4 + V-SEC ×4 + V-KEYMAT ×1 + 各轨道展开子项补足到 47；不足 47 时 Total 取实际并在报告诚实标注「精简集 N 项 / 基线 47」）。

**双态执行（诚实标注，对齐 Orchestrator 在线=done/离线=simulated 的诚实原则）**：
- **TLS 类用例真实探测**（V-PROTO-01/02、V-SEC-01/03、V-KEYMAT）：复用 `scan.TLSScanner` 思路，对 `Run.Target` 用 `crypto/tls` 握手——
  - V-PROTO-01：`tls.Config{CurvePreferences: [X25519MLKEM768]}` 握手成功且 `ConnectionState().CurveID == X25519MLKEM768` → pass；Go 1.23+ 原生支持该混合组。
  - V-PROTO-02：仅 `CurvePreferences:[X25519]` 握手成功 → pass（向后兼容）。
  - V-SEC-01：对「强制混合」端点剥离 PQC 仅发 X25519，**握手被拒/降级未达成** → pass（被正确拒绝才算通过）。
  - V-SEC-03：`InsecureSkipVerify:false` 下证书验证失败即连接中止 → pass。
  - V-KEYMAT：探测成功后断言派生链三要素——`X25519 ∥ ML-KEM-768` 拼接 → `HKDF-Expand-Label(HKDF-SHA-256)` → context 绑定 `hybrid-kem-v1`（对齐 draft-ietf-tls-hybrid-design），三要素齐全 pass，缺一 fail。探测态记录实测 CurveID 作证据；模拟态填基线期望值并标 `simulated`。
- **不可平台直探的用例**（V-PROTO-04 swanctl、V-PROTO-05 OCSP、V-PROTO-06 ML-DSA 双签、V-COMPAT 客户端矩阵、V-PERF 性能、V-SEC-02/04）：给 `simulate` 态，输出文件4基线实测值并显式标 `Evidenced=simulated`：
  - V-PERF 阈值门（FR-7.3）：握手 p99 +8.3ms（39.5 vs 31.2）`<15ms` → pass；RPS −6.5%（1720 vs 1840）→ **conditional**「建议开启会话复用」；IKEv2 +32ms（412 vs 380）`<15%` → pass。`MeasuredMs` 落实测值。
  - V-COMPAT（FR-7.2）：Chrome131+/FF132+/Android12+/VPN v5.2 → pass；VPN v4.x → conditional 挂 R-001 风格宽限计时；TLS1.2 遗留 Java → conditional 转 P3 挂 R-001。
  - V-SEC-02/04：重放 TLS1.2 ClientHello 被拒、虚假 OCSP 签名校验失败 → pass。

**Gate 判定公式（FR-7.6/1.8 KPI 锁定口径）**：
```
GatePass = (Failed == 0)
        && (Passed + Conditional == Total)
        && (Passed >= floor(Total * 44/47))   // 基线 44/47，精简集等比例下取整
        && (所有 Conditional 项 RiskRef != "")  // 有条件项必须挂遗留风险登记
P1 覆盖：SLO-07 = P1Covered/P1Total，签署前须 == 100%（16/16 基线）
```
仅 `GatePass && SignState 链路完整` 才允许 `sign`，置资产 `verified`。

**报告生成**（复用 `report` 包风格，新 `verify/report.go`）：8 段对齐文件4——① 验收总览（44通过/3有条件/0未通过 计数 + GatePass + P1 覆盖率）② 协议层逐项（V-PROTO 表）③ 兼容性矩阵 ④ 性能基准与阈值门 ⑤ 安全验证四向量 ⑥ 混合密钥材料溯源 ⑦ 遗留风险登记（R-001..004）⑧ 签署区（状态/哈希/签署人）。复用 `algoFamily`、表格 builder 写法、黏土橙基准锚点说明。

### 验收标准
1. `GET /verify/cases?track=tls-hybrid` 返回 ≥6 条协议层用例，字段含 probe 命令与 expect 断言。
2. 对一个真实可达 TLS 1.3 + X25519MLKEM768 端点建 `probe` 模式 Run，V-PROTO-01 实测 `CurveID==X25519MLKEM768` 判 pass，证据 `Evidenced=probe` 且含实测握手元数据。
3. 对仅支持 X25519 的端点，V-PROTO-01 fail、V-PROTO-02 pass（向后兼容正确区分）。
4. V-SEC-01 对强制混合端点剥离 PQC，握手被拒 → pass；未被拒 → fail 并标安全事件。
5. V-KEYMAT 三要素（X25519∥ML-KEM-768 拼接 / HKDF-Expand-Label·HKDF-SHA-256 / context=`hybrid-kem-v1`）齐全判 pass，人为缺一判 fail。
6. `simulate` 模式跑全套，输出 44 通过 / 3 有条件 / 0 未通过、P1 16/16，且所有 simulated 项在结果与报告中明确标「模拟」。
7. V-PERF 握手 p99 增量 8.3ms`<15ms` pass、RPS −6.5% conditional、IKEv2 +32ms`<15%` pass，`MeasuredMs` 落实测值。
8. Gate 判定：有条件项未挂 RiskRef 时 `GatePass=false`；全部挂接后 `GatePass=true`。
9. `POST /verify/runs/:id/report` 生成 md 含 8 段且总览计数与逐项一致；报告 Hash 为内容 SHA-256。
10. 签署流：DRAFT→submit→UNDER_REVIEW→approve→（GatePass 且 conditional 全挂）→sign→SIGNED；SIGNED 后关联资产 `Status==verified`；reject 回 REJECTED。GatePass=false 时 sign 被拒。
11. `POST /remediations/:id/verify` 对 status=done 工单建 Run 并启动，前端可从工单跳转到该 Run 看逐项。
12. 前端运行页轮询展示逐项 pass/有条件/fail 实时推进，判定 tag 配色为黏土橙暖色系，simulated 诚实标注可见。
13. `viewer` 角色访问 `POST /verify/runs` 被拒，`GET` 类放行。

## ① 发现深化 —— 30 规则库 + 多扫描器 + 调度/导入/手录

### 目标与价值

闭环首环「发现(Discover)」当前只有单一 TLS 主动扫描器（M1 子集），覆盖度不可度量、无规则可追溯、无法消化离线证据。本领域把 PRD 1-1 的 **30 条规则（L1=8/L2=8/L3=8/L4=6，极高 7、P1 高优 14）** 固化为内置规则库并在每条 `ScanResult` 上命中标注，扩出 SSH KEX 审计扫描器与导入/手录三类"非主动"发现入口，补上周期调度（给 ⑤监测复扫直接复用）。现在做的理由：CBOM 建档（②）的去重锚点 `asset_fingerprint`、评估（③）的 `risk_hint` 参考、监测（⑤）的复评回灌全部以"发现层证据契约"为唯一上游——发现层不补厚，下游四环的数据底座就是空的。

### 编排/算法/计算逻辑

**扫描器接口扩展**（最小改动，复用现有 `scan.Scanner`）：
```go
// 现有 Scanner 不变：Scan(ctx, host, port) (*model.ScanResult, error)
// 新增能力声明，供 Runner 按 ScannerType 选择并标注 Method/Layer
type Scanner interface {
    Scan(ctx context.Context, host string, port int) (*model.ScanResult, error)
    Method() string  // "M1"
    Name() string    // "tls" / "ssh"
}
```
- `TLSScanner` 补 `Method()="M1" Name()="tls"`，零行为变化。
- 新增 `SSHScanner`（`golang.org/x/crypto/ssh`，在 `HostKeyCallback` 里截获 server 协商的 KEX 算法与主机密钥类型；`Name()="ssh"`，默认 port 22）→ 命中 **R-L1-05**（SSH KEX）、**R-L2-05**（SSH 主机密钥）。
- `Runner.NewRunner` 按 `job.ScannerType` 装配；占位扫描器（IKE/RDP）只注册元数据返回"未实现"，规则库可见但不阻塞。

**命中引擎 `scan/rules.go`**（在 `Runner` 落 ScanResult 后调用，输入 ScanResult，输出 []RuleHit）：

| 规则 | 命中条件（机读直证→置信度=高） | RiskHint |
|------|--------------------------------|---------|
| R-L1-01 | `TLSVersion ∈ {TLS 1.0/1.1/1.2}` | 中 |
| R-L1-02 | CipherSuite/KEX 含 `RSA/ECDH/DHE`（非纯 PQC/混合） | 高 |
| R-L1-03 | `SigAlgo ∈ {RSA-PKCS1, ECDSA, Ed25519}` | 高 |
| R-L1-05 | SSH KEX ∈ 经典 DH/ECDH（curve25519 不命中） | 中 |
| R-L2-01 | 导入证书 `IsCA && Subject==Issuer && Algo∈{RSA,SM2}` | 极高 |
| R-L2-02 | 服务器证书 `Algo∈{RSA,ECDSA}` | 高 |
| R-L2-05 | SSH 主机密钥 `Algo∈{RSA,ECDSA}` | 中 |
| R-L4-01 | SBOM 含 OpenSSL<3.5/LibreSSL/mbedTLS 不支持 ML-KEM | 中 |
| R-L4-05 | SBOM 含外部依赖密码库组件 | 中 |

**置信度判定（FR-3.4.2）**：握手 KEX / 证书签名 OID / 库版本号 = `高`；mTLS 状态、SSE 算法等上下文推断 = `中`；M7 仅人工申报 = `低`。

**asset_fingerprint（FR-3.10 去重锚点）**：`sha256(lower(host)+":"+port+"|"+service+"|"+certFingerprint)`，证书类无 host:port 时退化为 `sha256("cert|"+certfp)`。同指纹在②建档合并为 1 资产、保留多条 RuleHit 证据来源。

**增量模式（FR-3.5.5）**：`mode=incremental` 时先查同 fingerprint 的 `LastSeen`，握手特征无变化只更新 `LastSeen` 不新建 result。

### 验收标准

7. 人工录入资产 `source=manual`、`confidence=低`、`evidenceNote` 缺失被拒入库（FR-3.3.6/3.8.1）。
8. 给任务设 cron 后 `nextRunAt` 被刷新；到点自动复跑并更新 `lastRunAt`，监测可复用同 job（FR-3.5.2）。
9. 同一 host:443 证书被 M1 与 PEM 导入两次发现，`assetFingerprint` 相同，建档侧合并 1 资产保留 2 条 RuleHit 证据（FR-3.1/3.10）。

### 编排/算法/计算逻辑

#### 1. 引擎扩展（复用核心，不重写）
`scoring.Score(d Dimensions)` 现以包级常量 `WeightD1..D5` 加权。新增**可传权重**重载，原函数保留为默认权重的薄封装以不破坏既有调用方：

```go
// Weights 五维权重（百分数，Σ=100）。
type Weights struct{ W1, W2, W3, W4, W5 int }

// StandardWeights 标准锁定权重 30/25/20/15/10（DP-11 基线）。
var StandardWeights = Weights{30, 25, 20, 15, 10}

// ScoreWith 以自定义权重计分；其余分级/HNDL/窗口口径与 Score 完全一致。
func ScoreWith(d Dimensions, w Weights) Result {
    raw := (float64(d.D1)*float64(w.W1) + float64(d.D2)*float64(w.W2) +
            float64(d.D3)*float64(w.W3) + float64(d.D4)*float64(w.W4) +
            float64(d.D5)*float64(w.W5)) / 100.0
    score := roundHalfUp(raw)
    level, text, window := classify(score)
    return Result{Score: score, RawScore: raw, Level: level, LevelText: text,
        Window: window, HNDL: d.D2 >= 60 && d.D3 >= 60}
}
// Score 保持签名不变，转调 ScoreWith(d, StandardWeights)。
```

**关键不变量（须单测断言）**：
- `classify` 阈值不随权重变：P1≥75 / P2≥50 / P3≥25 / P4<25。
- **HNDL 与权重解耦**：恒为 `D2≥60 ∧ D3≥60`，调权不影响 HNDL 判定（PRD 口径，避免调权稀释 HNDL 专项）。
- 权重校验：`W1+W2+W3+W4+W5==100` 且各 `≥0`，否则 API 400；标准方案值必须等于 `StandardWeights`。
- `ScoreWith(d, StandardWeights)` 对任意 d 与旧 `Score(d)` 结果逐字段相等（回归保护）。

#### 2. 激活复算状态机 / before-after 对比
`activate` 与 `rescore` 共用复算流程，事务内执行：
1. 快照当前生效方案下全资产等级分布 `before{P1..P4}`。
2. 逐资产 `ScoreWith(dims, newWeights)` → 更新 `CryptoAsset` 评分字段 → 写 ScoreHistory（`PrevScore/PrevLevel` 取该资产上一条快照）。
3. 累计 `shift["P2->P1"]++` 等级迁移矩阵；统计 `after{P1..P4}`。
4. 落 `RescoreRun`，返回 `{before, after, shifts, runId}`。

`preview` 走相同计算但**只读不落库不切 IsActive**，用于面板实时预演。

#### 3. recompute 接生效方案
`assets.go` 的 `recompute(a)` 改为读当前 `IsActive` 方案权重调 `ScoreWith`（无生效方案则回退 `StandardWeights`），保证单资产手工评分与批量复算口径一致。

### 编排/算法/计算逻辑

**1) 资产归一指纹 `assetDigest`（diff 与去重共用）**
对每个 CUP 计算稳定 digest，落入快照 `DigestJSON`：
```
digest = sha256( algorithm | keySize | protocol | certFingerprint | status | riskLevel )
key    = certFingerprint 非空 ? "fp:"+certFingerprint
       : endpoint 非空      ? "ep:"+endpoint+"|"+protocol
       : "nm:"+name
```
`key` 为跨快照配对的稳定锚（对齐 PRD 4.3 去重主键优先级 ①②⑤），`digest` 变化即「内容变更」。

**2) 快照 diff 算法（FR-4.6.2 六类输出）**
对 `from`、`to` 两侧按 `key` 做集合配对：
- `key ∈ to 且 ∉ from` → **新增(added)**
- `key ∈ from 且 ∉ to` → **移除(removed)**
- `key` 两侧都在但 `digest` 不同 → 细分变更类型，按字段比较：
  - `algorithm` 变化 → **算法变更(algo_changed)**；若新算法 ∈ {ML-KEM-768, ML-DSA-65, X25519+ML-KEM-768, SLH-DSA…} 标 `isProgress:true`（迁移进展）
  - `certFingerprint` 变化 → **证书续期/更换(cert_rotated)**
  - `status` 变化 → **状态变化(status_changed)**
  - `riskLevel` 变化 → **风险等级变化(level_changed)**，带方向（↑恶化/↓改善）
- **算法分布环比**：`algoDistDelta = to.AlgoDist − from.AlgoDist`，输出每算法 count 增减（如 `RSA: -3` 体现占比下降，对齐 4.6.2 完成定义）。

响应结构：
```json
{
  "from": {"id":1,"name":"2026Q1-baseline"},
  "to":   {"id":0,"name":"实时"},
  "summary": {"added":4,"removed":1,"algoChanged":6,"certRotated":2,"statusChanged":3,"levelChanged":2},
  "added": [{"name":"...","algorithm":"RSA-2048","key":"fp:..."}],
  "removed": [...],
  "changed": [
    {"name":"vpn-gw:4500","key":"ep:...","type":"algo_changed",
     "from":"ECDH","to":"X25519+ML-KEM-768","isProgress":true}
  ],
  "algoDistDelta": {"RSA":-3,"ML-KEM-768":4,"ECDH":-2}
}
```

**3) 多源合并置信度升级**（FR-4.3.1）：合并后 `distinctScanners = COUNT(DISTINCT scanner)`；`≥2 ⇒ confidence=高(100)`，单来源保持原值，算法冲突不静默丢弃，进「冲突待裁决」（`RiskHint` 标记 + 前端红标）。

**4) 状态机合法迁移表**（FR-4.4.1，硬编码白名单）：
```
discovered → {confirmed, merged, archived}
confirmed  → {archived, merged}
archived   → {confirmed}            // 重新激活（"已忽略非终态"）
merged     → {}                     // 终态
```
不在白名单的迁移一律 422，对齐「尝试 discovered 直接到 verified 被拒绝」。

> 合并/状态/CBOM 导入后若算法字段变化，**复用 `recompute(&a)`**（assets.go 现成）重算五维与 HNDL，保证评分一致。

### 目标与价值
当前后端仅有 JWT 单 `admin`，所有 `/api/v1` 端点同权、无写操作约束、敏感动作无任何留痕，与 PRD 第 10 章 RBAC 6 角色与「可审计」诉求差距明显。本领域补的是贯穿五阶段闭环的**治理底座**：把已存在的 `User.Role`（admin/operator/viewer 常量已定义但从未在中间件消费）真正接入路由授权、给扫描/评分/改造/设备/导出等敏感操作落审计日志、提供用户与系统设置入口、并用时间序列快照让仪表板从「当前快照」升级为「迁移进度趋势」。它不新增业务环，而是让已有的发现/评估/改造各环可管、可控、可回溯。

### 编排/算法/计算逻辑

**RBAC 矩阵（角色 × 模块 × 操作）** — 中间件按此判定。本期落地三角色（PRD 6 角色为远期，此处实现 admin/operator/viewer 并预留扩展位）：

| 模块 \ 角色 | viewer | operator | admin |
|---|---|---|---|
| 读（所有 GET：dashboard/assets/scans/score/reports/devices/playbooks/remediations/settings/audit） | ✓（audit 除外） | ✓ | ✓ |
| 扫描发起 `POST /scans` | ✗ | ✓ | ✓ |
| 资产增改删、评分 `POST/PUT/DELETE /assets*`、`/assets/:id/score` | ✗ | ✓ | ✓ |
| CBOM 导出 `/cbom/export` | ✗ | ✓ | ✓ |
| 报告生成 `POST /reports` | ✗ | ✓ | ✓ |
| 设备增改删/连通 `POST/PUT/DELETE/test /devices*` | ✗ | ✓ | ✓ |
| 改造建单/执行/回滚 `POST /remediations*` | ✗ | ✓ | ✓ |
| 快照触发 `POST /metrics/snapshot` | ✗ | ✓ | ✓ |
| 审计查询 `GET /audit-logs` | ✗ | ✓（只读） | ✓ |
| 审计导出 `/audit-logs/export` | ✗ | ✗ | ✓ |
| 用户管理 `/users*` | ✗ | ✗ | ✓ |
| 设置写 `PUT /settings/:key` | ✗ | ✗ | ✓ |

**授权中间件状态判定（伪逻辑）**：
```
requireRole(min...): 读 c.Get("role")；不在允许集合 → 403 {"error":"权限不足"} 并写 AuditLog{Result:"denied"}
```
路由分组策略：在 `router.go` 现有 `auth` 组下再拆两个子组 `writer := auth.Group(""); writer.Use(s.requireRole(Operator, Admin))` 与 `adminGrp := auth.Group(""); adminGrp.Use(s.requireRole(Admin))`，把写端点迁入对应子组（读端点留在 `auth`）。`authMiddleware` 额外查 `User.Status==active`，禁用用户即时失效。

**审计落库**：实现 `s.audit(c, module, action, target, result, detail)` 辅助函数，从 `c.Get` 取 actor 快照、`c.ClientIP()` 取 IP，异步或同步写 `AuditLog`（同步即可，单机 SQLite 量级小）。在每个写 handler 成功/失败分支调用。

**动作字典（枚举常量）**：`scan.create`、`asset.create/update/delete/score`、`cbom.export`、`report.create`、`device.create/update/delete/test`、`remediation.create/execute/rollback`、`user.create/update/delete/password`、`setting.update`、`auth.login`（成功与失败都记，失败用于暴力破解审计）。

**权重方案生效路径**：`scoring.weights` 设置仅作为「当前生效方案」的元数据存储与展示口径；当用户改权重并对资产重新评分时，评分入口读取该设置覆盖 `scoring` 包默认常量（通过给 `scoring.Score` 增加 `ScoreWith(d, weights)` 重载，保留无参 `Score` 用默认值向后兼容，PRD 默认 D1×0.30…D5×0.10、P1≥75 仍是 seed 与回退值）。**本期 P0 仅做存储+展示+校验**，重打分按当前权重为 P1 子项。

### 验收标准
1. operator/viewer 登录后，调用任一写端点（如 `POST /scans`、`DELETE /devices/:id`）：viewer 必返回 403 且前端不显示对应按钮；operator 可写但访问 `/users` 返回 403。
2. admin 可在「用户管理」页新建 operator 用户，新用户能登录、读取全部页面、发起扫描，但看不到「用户管理」「系统设置」菜单。
3. 禁用某用户后，其现有 token 下一次请求即被拒（401/403），无需等 token 过期。
4. 删除/降级最后一个 admin 被拒（409，前端 toast 明确提示）。
5. 完成一次扫描、一次评分、一次改造执行、一次 CBOM 导出后，`/audit-logs` 能查到对应 4 条记录，含正确的 actor/action/module/target/result/IP/时间；登录失败也产生 `auth.login + failure` 记录。
6. 审计页过滤（按 module=remediation、result=denied、时间范围）返回正确子集；CSV 导出文件可下载且字段完整。
7. 设置页改权重使五维和≠1 时保存被禁用并告警；改为合法值保存后 `GET /settings` 反映新值，`scoring.weights.preset` 标记为 `custom`。
8. SLO 阈值页显示 PRD 默认值（0.1% / 8.3ms / 15ms），「恢复默认」可一键回填。
9. `POST /metrics/snapshot` 后 `/dashboard/trend?days=7` 至少返回 1 个数据点；连续多日（或多次手动采集不同日期）后折线正确呈现资产总数/P1/已改造三条序列。
10. 所有新端点未带 JWT 一律 401；角色不足一律 403 并落 `denied` 审计。

```
⑥RBAC/审计中间件 ──(actor上下文/写权限)──▶ 所有写端点(①②③④⑤)
③ScoreWith(可传权重) ──(复算口径)──▶ ②合并/导入重算 · ④复评 · ⑤情报/漂移复评
②AssetEvidence(证据链) ──(命中证据来源)──▶ ①RuleHit · ④有条件挂接 · ⑤漂移证据
①scan.Scheduler(cron ticker) ──(同一进程内调度入口)──▶ ⑤监测复扫(Monitor 复用同机制)
②CbomSnapshot ──(快照/diff/AlgoDist)──▶ ⑤监测 CBOM 快照 diff(同一能力)
⑥MetricSnapshot/computeMetrics ──(时序聚合)──▶ ⑤趋势/漂移趋势(同源口径)
④AcceptanceRun(GatePass·SIGNED) ──(状态机出口)──▶ R2 工单→「已验收」· ⑤监测入口(基线/复评队列)
⑤MonitorEvent.reassess · ④情报复评 ──(J5→J2回灌)──▶ ③scoring.Score 重评分
```

#### 必须收口的冲突 / 重叠（关键决策）

| # | 冲突点 | 6 份规格各自的写法 | 裁决（单一真相源） |
|---|---|---|---|
| C1 | **调度器**：① `scan.Scheduler` 与 ⑤ `monitor.Scheduler` 都建进程内 cron ticker | 各起一个 | **合一**：建 1 个进程内调度框架（`internal/sched` 或 `scan.Scheduler` 升格），①周期扫描与⑤监测复扫都注册到同一 ticker。避免双 ticker/双 cron 解析依赖。Wave 内由①先建、⑤复用。 |
| C2 | **CBOM 快照/diff**：② `CbomSnapshot` 与 ⑤ `CBOMSnapshot` 是两套模型+两套 diff | ②建 `internal/api/snapshot.go`；⑤建 `internal/monitor/snapshot.go` | **以②为准**：快照模型+diff 算法+digest/key 归一只在②实现一次；⑤监测域**调用②的快照与 diff**（rescan 后触发 `cbom.Build`+建快照+调 diff），不另立表、不另写 diff。 |
| C3 | **权重可传化**：③ `scoring.ScoreWith(d,Weights)` 与 ⑥「权重存 Setting + 重打分」 | ③建 `Weights/ScoreWith/ScoreProfile`；⑥也想存 `scoring.weights` 到 Setting 并重载 | **以③ ScoreProfile 为权重唯一真相源**；⑥的 `scoring.weights` 设置项**降级为只读展示/回退默认值**，不作第二套生效路径（避免 Setting 与 ScoreProfile 双写打架）。⑥引用③的 `ScoreWith`。 |
| C4 | **遗留风险登记 R-00x**：④ `verify/risks` 与 ⑤ `LegacyRisk` 都种子 R-001..004、都做台账 | ④ `GET /verify/risks`；⑤ `LegacyRisk` 模型+CRUD | **以⑤ `LegacyRisk` 为准**（字段更全：AlwaysOnSLO/RemediationID/RecheckDate）；④的有条件项 `RiskRef` **挂接到⑤的 LegacyRisk**。④只读引用，不重复建表/种子。 |
| C5 | **复评回灌入口（J5→J2）**：④（签署后置 verified）、⑤（漂移/情报→复评）、③（rescore） | 各自调 `scoring.Score` | **统一走③的 rescore 管线**：④/⑤产生「复评信号」→ 调③ `recompute`/`ScoreWith`（当前生效方案）并写 `ScoreHistory(Reason=reassess)`。复评只有③一条真实重算路径。 |
| C6 | **资产去重锚点 `assetFingerprint`**：①（发现层契约）与②（建档去重主键） | ①定义 `sha256(host+port+service+certfp)`；②定义优先级 cert→endpoint→name | **统一 fingerprint 公式与去重主键优先级**：以②的去重主键优先级为权威，①产出的 fingerprint 字段值按②公式生成。一处函数 `assetFingerprint()`，①②共用。 |
| C7 | **资产状态机**：②扩 `merged/inventoried`、④置 `verified`、⑤置回 `discovered/已评分`、R2 走 `accepted/monitored` | 各加状态 | **建一张全局状态机白名单表**（②为主笔），④的 `verified`、⑤的复评回退、R2 的改造态全部并入同一张迁移白名单，任何域不得私开非法迁移。 |
| C8 | **证书数据来源**：⑤证书到期预警用 `CryptoAsset.CertNotAfter`（R1 已采集） | ⑤直接读 | 无冲突，确认：⑤不新增证书扫描，①PEM 导入补齐 `CertNotAfter` 即可被⑤消费。 |

#### 共享构件清单（建一次，多域复用）

- **异步执行范式**：`scan.Runner`/`remediate.Orchestrator` 的「同步 `Run(ctx,id)`+调用方 goroutine+逐步落库轮询」→ ④`verify.Runner`、⑤`monitor.Runner` 照搬。
- **JSON 列辅助**：`db.MarshalStrings/UnmarshalStrings`、`MarshalEvidence`、`Device.Capabilities` 的 `gorm:"-"`+镜像字段模式 → 全域新模型套用。
- **进程内调度**：C1 合一的单 ticker → ①周期扫描 + ⑤监测复扫。
- **CBOM 快照+diff**：C2 收口到②一套 → ⑤复用。
- **评分引擎**：③`ScoreWith`/`classify`/`roundHalfUp`/`Derive`/`SuggestAlgo` → ②④⑤⑥全部复用，零重造。
- **CBOM 导出**：R1 `cbom.Build/Export` → ②快照、⑤快照内容序列化复用。
- **RBAC/审计**：⑥`requireRole`+`s.audit()` → 所有域写端点统一挂载。
- **TLS 探测**：R1 `scan.TLSScanner`（`crypto/tls`/`ConnectionState`/`CurvePreferences`）→ ①SSH/导入、④协议层探测、⑤复扫漂移复用。

### 2) 三个施工波次

#### Wave A — 补齐闭环 + 最高 ROI（与 R1/R2 衔接最紧）

**为何此顺序**：R2 把改造工单跑到 `done`，但 PRD 状态机硬约束「`done`→`已验收`必须过验收套件」——没有④，任何改造工单无法合法流转，闭环在第④环出口断裂；⑤监测是闭环第⑤环、是「已验收→监测中→复评」的最后一块，且其复评回灌让闭环成环。同时③的 `ScoreWith` 是④/⑤复评的共同前置（C3/C5），⑥的 RBAC/审计是所有写端点的 actor 底座（必须先于大量新写端点落地，否则后补埋点成本翻倍）。因此 Wave A = **闭环出口（④）+ 闭环成环（⑤核心）+ 二者共同前置（③引擎可传权重 + ⑥治理底座）**。

**包含子项**（精确范围见第 3 节）：
- **③-引擎层（P0 子集）**：`scoring.ScoreWith(d,Weights)`+`StandardWeights`+回归单测；`Score` 转调；`recompute` 接生效方案；`ScoreProfile`/`ScoreHistory` 模型+内置标准方案 seed。（C3/C5 前置，**不含**前端调权面板/登记册——留 Wave B）
- **⑥-治理底座（P0 子集）**：`requireRole` 中间件+路由分组（写操作限 operator/admin）；`authMiddleware` 加 `User.Status` 校验；`AuditLog` 模型+`s.audit()`+写 handler 埋点；`User` 扩字段+用户管理 API。（**不含**Settings/趋势——留 Wave B/C）
- **④-验收自动化（全 P0+签署）**：`AcceptanceRun/TestResult/AcceptanceReport`+静态 cases；`verify.Runner` 双态执行+Gate 公式；TLS 真实探测（V-PROTO-01/02、V-SEC-01/03、V-KEYMAT）；API（cases/runs/report/sign+`/remediations/:id/verify` 接缝）；前端 `Acceptance.vue`。
- **⑤-监测核心（P0 子集）**：6 模型+AutoMigrate+R-00x/默认策略 seed；调度器（**复用 C1 合一 ticker**）；`monitor.Runner` 复扫+漂移检测+SLO-05 P1 告警；复评回灌（→③重评分，C5）；证书到期分级预警；MonitorPolicy/Event/LegacyRisk CRUD+处置 API。（SLO 时序/情报/快照 diff/前端仪表板留 Wave B）

**产出的用户可见价值**：改造工单 `done` 后可**一键跑验收 → 逐项 pass/有条件/fail → 生成可签署报告 → 签署后资产置「已验收」**；已验收资产进入**监测**，混合算法回退/证书临期被检出并**自动开复评批次回流评估页**——闭环从「一次性项目」首次跑通为「改造→验收→监测→复评」可循环治理。同时全平台写操作受 RBAC 约束、敏感动作留审计。

**退出标准**：
1. `go build ./...` 通过；新模型 AutoMigrate 无破坏；空库 seed 出标准权重方案 + R-001..004 + 默认季度策略。
2. `ScoreWith(d,StandardWeights)` 对种子 7 资产与旧 `Score(d)` 逐字段相等（回归单测绿）；HNDL 任意权重恒 `D2≥60∧D3≥60`。
3. `POST /remediations/:id/verify` 对 `done` 工单建 Run；`simulate` 全套出 44 通过/3 有条件/0 未通过、P1 16/16，simulated 项诚实标注；TLS `probe` 端点 V-PROTO-01 判定正确。
4. 签署链 DRAFT→submit→approve→sign→SIGNED，SIGNED 后资产 `Status==verified`；GatePass=false 时 sign 被拒。
5. 漂移闭环：已迁混合端点复扫得纯经典 → 30s 内 `severity=p1` `drift` 事件 + SLO-05 告警 + 自动复评批次（资产回流评估页可见重评分）。
6. 证书预警：CA 剩 170 天 / 服务器剩 25 天分级正确触发。
7. RBAC：viewer 调任一写端点 403 并落 `denied` 审计；operator 访问 `/users` 403；删/降最后一个 admin 被拒。
8. 完成一次扫描/评分/改造/验收后 `/audit-logs` 可查到对应记录（actor/action/result/IP）。

#### Wave B — 数据底座做厚 + 评估可信化 + 监测可视化

**为何此顺序**：Wave A 跑通了闭环骨架，但「喂给闭环的数据」仍单薄——①发现只有单扫描器/无规则可追溯、②建档只读不可合并、③评估不可调权不可追溯。Wave B 把**数据入口与可信度**补厚：①30 规则库让发现可度量、②合并/证据链让建档归一为唯一 CUP（这是④有条件挂接、⑤漂移证据的上游 C4/C6），③调权面板+历史让评分可审计上报。⑤的 SLO 时序/前端仪表板依赖 Wave A 的事件模型已就位，此波补全可视化。顺序上①②先于③消费（资产先归一，评分才有意义）。

**包含子项**：
- **①发现深化**：`ScanRule`+`RuleHit` 模型+30 条 seed；命中引擎 `scan/rules.go`；`RuleLibrary.vue`+Discovery 命中标注；Scanner 接口扩展（Method/Name）；PEM/SBOM 导入端点；人工录入校验；`CoverageMatrix.vue`+`/coverage`；SSHScanner。
- **②建档深化（P0+P1）**：资产 CRUD 表单抽屉+状态机迁移校验（**并入 C7 全局白名单**）；多源去重合并 `/assets/merge`+`/dedup-candidates`+`AssetEvidence`+`MergeModal.vue`；证据链下钻+hash 固化；CBOM 导入 `cbom.Parse`+round-trip；`CbomSnapshot` 模型+create/list/export（**C2 唯一实现**）。
- **③评估深化（P0 余项+P1）**：`/score/profiles` CRUD+activate（全量复算+before/after）；`/assets/:id/history`；专家模式调权面板（实时分布预演）；评分历史时间线；`/score/rescore` 批量复算；`RescoreRun`+shift 矩阵；风险登记册+CSV 导出。
- **⑤监测可视化（P1）**：SLO 时序 ingest/series/summary+越界连续窗口判定；威胁情报订阅（CRUD+拉取+复评映射，**复用 C5/④风险台账 C4**）；`Monitor.vue` 仪表板（SLO 卡+告警台账+证书甘特）；SLO 趋势图组件；`LegacyRiskTable`/`ThreatIntelFeed`。

**产出的用户可见价值**：发现端可浏览 30 条规则库、看每条扫描结果命中哪条规则、用 PEM/SBOM/手录补充资产、看 L1-L4×M1-M7 覆盖度；建档端可新增/编辑/合并资产、下钻原始证据链、按去重建议归一；评估端可开专家模式调权并实时预演 P1-P4 分布、看资产评分历史时间线、导出风险登记册；监测端有完整仪表板（SLO 八卡+告警台账+证书到期甘特+情报流+一键复评）。

**退出标准**：
1. `GET /rules` 恰 30 条、统计头 `total=30/p1High=14/critical=7/byLayer`；「仅极高」筛出 7 条；TLS 端命中 R-L1-01/02/03 带 evidence。
2. 同一 host:443 证书被 M1 与 PEM 导入两次，`assetFingerprint` 相同，合并为 1 CUP 保留 2 条 RuleHit/Evidence（C4/C6 验证）。

### 4) 全局风险与必须遵守的既有口径（所有 Wave 强制）

#### 既有口径（红线，不可偏移）

- **五维权重**：D1×30 + D2×25 + D3×20 + D4×15 + D5×10（Σ=100）；`StandardWeights{30,25,20,15,10}` 为内置只读基线，调权须留痕+全量复算+版本对比（DP-11）。权重唯一真相源 = ③ `ScoreProfile`（C3）。
- **分级阈值**：P1≥75 / P2≥50 / P3≥25 / P4<25；`classify` 阈值**不随权重变**（单测断言）。
- **HNDL**：恒为 `D2≥60 ∧ D3≥60`，**与权重解耦**，任何调权不得稀释 HNDL 判定。
- **SLO 阈值**（监测/验收/设置默认必须锚定）：握手失败率 0.1%（SLO-01）；延迟增量 +8.3ms、阈值 15ms（V-PERF/SLO-02，p99 基线 39.5 / 阈值 46.2）；吞吐降幅 6.5%；IKEv2 +32ms 上限 437；CBOM 新鲜度 90 天；CA 预警 180 天。验收基线 = 44 通过/3 有条件/0 未通过、P1 覆盖 16/16。
- **混合算法写法**：统一 `X25519+ML-KEM-768`（或 `X25519MLKEM768` for CurveID）、`ML-DSA-65`、`SLH-DSA`；KEM 派生链 `X25519 ∥ ML-KEM-768` → `HKDF-Expand-Label(HKDF-SHA-256)` → context `hybrid-kem-v1`（draft-ietf-tls-hybrid-design）。检出「混合→纯经典回退」即 P1+复评（SLO-05 目标 0）。
- **黏土橙暖色系**：accent `#B4552D`/`#DB855C`，Hanken Grotesk；三级严重度 P1 红 / 预警橙 `#DB855C` / 巡检黄；所有新前端页严格用 `--clay-*` token，不得引克莱因蓝。
- **技术栈不漂移**：Go(Gin+GORM+glebarez 纯 Go SQLite+JWT, :8099) / Vue3+TS+Vite+Arco(:5390)；**不引入新第三方重依赖**（cron 用进程内 ticker、图表优先轻量 SVG）；离线私有化（NFR-8.7.1），无外部依赖、无 5xx，合成数据诚实标 `synthetic`。
